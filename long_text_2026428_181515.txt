# 《协同文档下的 Agent 协作闭环》分享说明

## 分享主题

### 标题

协同文档下的 Agent 协作闭环

### 一句话主题

当 Agent 真正进入多人协同文档、开始修改真实业务产物时，系统怎样把它的行为接住，让它变得可见、可判断、可撤回的一次实践。

### 核心表达

这里的“接住”，具体包括：

- 用户能看见它改了哪里
- 用户能判断这次修改值不值得接受
- 用户能在协同场景里精确撤回 AI 的这一轮修改
- 系统能把这轮修改沉淀成可比较、可复盘的事实

## 分享大纲

### 1. 问题从哪里开始

从个人使用 AI、做 Agent、再回到业务场景的观察切入，提出一个核心问题：

AI 会生成已经不稀奇，真正难的是它进入真实业务产物之后，系统能不能承接它。

### 2. 当 AI 进入协同文档

把问题具体化到协同文档场景：

- 文档不再只是人与人的协作
- 还多了一个高速、大规模、异步的编辑者
- 这时风险不只是“写得好不好”，而是会不会破坏信任和掌控感

### 3. 可信协作需要哪些设计

这部分是分享主体，会讲当前设计中的几个关键点：

- Agent 以什么身份参与协作
- `Ctrl+Z` 和“撤回 AI 这一轮”为什么要分开
- 为什么透明化编辑很重要
- 事实链怎样支撑 `message-diff` 和 `run-diff`
- Diff 为什么不是越细越好，而是越能帮助判断越好
- 记录层、展示层、决策层为什么缺一不可

### 4. 当前做到哪里

明确当前落地边界：

- 现在主要跑通的是前端侧的记录、Diff、撤回闭环
- 事实链当前主要用于支撑对比和撤回
- 这一步解决的是“看得见、判断得了、退得回”
- 但它还不是完整服务端版本系统

### 5. 后续可以往哪里演进

- 如果产品继续往知识工作台、项目空间、NotebookLM / YouMind 一类方向发展
- 就需要建设更完整的 I-P-O 模型
- 也需要把 run 级事实链、版本空间、恢复能力逐步沉淀出来

这里会强调：

- `Input` 是资料、上下文、用户目标
- `Process` 是 Agent 的处理过程、步骤和中间状态
- `Output` 是最终生成或修改出来的内容

### 6. 最终结论

收束到一个业务判断：

好的 Agent 系统，不是把结果塞给用户就结束，而是让 AI 可以大胆帮忙，同时让人始终掌控。

## 目标听众

这次分享适合以下几类听众：

- 在做 AI 产品、Agent 产品、Copilot 产品的产品经理
- 在做编辑器、协同、Diff、回放、撤回相关能力的前后端研发
- 在做业务平台、工作台、知识库、文档系统的技术负责人
- 对“AI 如何真正进入业务系统”感兴趣的同学

不太适合的听众：

- 只想听模型选型、Prompt 技巧、单次生成提效的人
- 希望看到完整 NotebookLM 产品拆解的人

因为这次分享的重点不是模型能力本身，而是业务系统如何承接 Agent 行为。

## 听众收益

听完之后，听众 会获得四类收获。

### 1. 一个更准确的问题框架

不再只从“AI 能不能生成”看 Agent，而是开始从下面这些问题评估它：

- 它以什么身份进入系统
- 它的修改能不能被看见
- 它的行为边界能不能被判断
- 它的改动能不能被精确撤回
- 它的过程能不能被系统沉淀

### 2. 一套可复用的设计视角

即使不是做协同文档，也可以把这套视角迁移到别的业务 Agent 里：

- 身份
- 边界
- Diff
- 反馈
- 审批
- 回放
- 撤回

### 3. 对当前实现边界的清晰理解

听众会知道哪些是当前已经做出来的，哪些是后续演进方向：

- 当前：`message-diff` / `run-diff`、前端记录、前端撤回、前端闭环
- 后续：run 级事实链、服务端版本空间、I-P-O 工作台能力

这有助于避免把当前方案讲得过高，也能帮助团队更清楚地讨论下一步投入值不值。

### 4. 一个更现实的 Agent 产品判断

听众会更容易接受这样一个事实：

Agent 是否可靠，不主要取决于模型是不是“更聪明”，而取决于系统有没有为它准备好承接、校验、撤回和恢复机制。

## 适合作为对外介绍的话术

这次分享会围绕“协同文档下的 Agent 协作闭环”展开。重点讲当 Agent 真正进入多人协同文档、开始修改真实业务产物时，系统怎样把它的行为接住，让它变得可见、可判断、可撤回。我们也会把当前实现边界和未来演进方向讲清楚：现在主要跑通的是前端侧记录、Diff、撤回闭环；如果未来产品继续往知识工作台或更完整的文档 Agent 方向发展，后续才需要进一步建设 run 级事实链、服务端版本空间和 I-P-O 模型。
