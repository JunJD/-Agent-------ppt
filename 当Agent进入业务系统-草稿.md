# 当 Agent 进入业务系统

> 一份面向 PPT 的内容草稿。  
> 目标不是解释“AI 很强”，而是讲清楚：Agent 如何从代码世界进入真实业务系统，以及业务系统为了接纳 Agent 需要重新设计哪些边界。

## 内容边界

这份 PPT 先聚焦四件事：

- 我与 AI：从模型入口到掌控 Agent
- 控制空间：为什么 Coding Agent 先成立
- 设计实践：回到协同文档项目，看 Agent 怎么落地
- 留在台上：尽情使用它，探索它的边界

暂时不展开：

- 大模型底层原理
- 各家模型能力对比
- 复杂的多 Agent 架构
- 完整商业化方案

## 0. 开场页

### 当 Agent 进入业务系统

副标题可选：

> 从 Coding Agent 到业务协作系统的一次产品设计思考

开场表达：

> 过去我们把 AI 看成一个回答问题的工具。  
> 但 Agent 出现以后，AI 不再只是生成内容，而是开始理解任务、调用工具、参与流程，并对业务状态产生真实影响。  
> 当它进入业务系统，问题就从“它聪不聪明”变成了“它能不能被组织安全、稳定、可控地使用”。

# 1. 我与 AI

> 从模型入口到掌控 Agent

## 1.1 为什么是我来讲这个话题

页面要点：

- 我较早开始接触 AI，也长期把 AI 放进真实工作流里使用
- Coding Agent 是我最早深度感受到的 Agent 形态
- 在代码世界里，Agent 已经不只是补全工具，而是可以读代码、改代码、跑测试、解释系统
- 我对 AI 的兴趣不是“工具尝鲜”，而是持续观察它如何改变人的工作方式

讲稿提示：

> 我最早感受到 Agent 不是在办公软件里，而是在代码世界里。  
> Coding Agent 很特殊，因为代码本身有明确的结构、文件、测试、版本控制和回滚机制。  
> 这让 Agent 可以比较早地进入一个真实工作环境。  
> 但也正因为这样，我开始意识到：如果 Agent 要进入更复杂的业务系统，它需要的不只是模型能力，还需要系统边界。

## 1.2 不同阶段的不同局限性

页面要点：

- 第一阶段：AI 像搜索和问答，能回答，但不理解业务现场
- 第二阶段：AI 像助手，能生成内容，但结果需要人复制、粘贴、判断
- 第三阶段：Agent 开始调用工具，但容易变成黑盒执行
- 第四阶段：业务 Agent 需要被嵌入权限、流程、数据、审计和协作关系中

讲稿提示：

> 每个阶段的问题都不一样。  
> 问答阶段的问题是“不知道上下文”。  
> 生成阶段的问题是“结果停留在文本里”。  
> 工具调用阶段的问题是“它真的开始做事了，但我们未必知道它为什么这么做”。  
> 到业务系统阶段，真正的问题变成了：谁授权它做？它能改什么？错了怎么办？过程怎么复盘？

## 1.3 一个核心判断

页面要点：

> 我在代码世界里学到的，不是“Agent 可以替人写代码”。  
> 而是：只要一个系统有清晰的边界、验收点和恢复能力，Agent 就有机会承担真实执行。

引出第二部分：

> 很多程序员其实已经和我一样，全面拥抱 Agent 了。  
> 这不是因为程序员更愿意冒险，而是因为代码世界天然有一套控制系统。  
> 所以下面我们先认识一下：代码世界是怎么接住 Agent 的，它依靠哪些控制系统，又形成了哪些体系语言。

可以强调：

- 聊天框解决的是交互入口
- Agent 解决的是任务执行
- 业务系统关心的是状态变化
- 组织关心的是责任边界
- 第二部分要讲的，就是代码世界为什么先给 Agent 提供了一个稳定工作的工程空间

# 2. 控制空间

> 为什么 Coding Agent 先成立

## 2.1 Coding Agent 为什么先成立

页面要点：

> Coding Agent 跑得快，不是因为代码简单，而是代码世界有一套现成的控制系统。

代码世界至少有四个好处：

- 看得见：文件、目录、依赖、diff、提交记录都摆在那
- 圈得住：大部分任务都在一个仓库、一个分支、一次 PR 里发生
- 能验证：typecheck、test、build、CI、review 都能帮忙验
- 能退回：分支、revert、tag、回退发布这些机制本来就成熟

讲稿提示：

> 今天看 Agent 产品，会发现最先跑出来的基本都是 Coding Agent。  
> 表面上看，好像是因为代码更适合大模型。  
> 但我觉得重点不在这里。  
> 代码一点也不简单，复杂度很高。  
> 只是代码世界早就有一套工程架子：改了哪里能看见，改得对不对能验证，错了还能退。  
> 所以 Agent 在这里不是裸奔，它是被仓库、diff、测试、review 这套东西兜住了。  
> 这就是 Coding Agent 能先成立的原因。

## 2.2 业务为什么复杂

页面要点：

> 业务复杂，不是因为它听起来高级，而是很多东西一开始就很空。

一个业务问题里，经常有三件事没说清楚：

- 用户到底想要什么
- 系统应该给什么
- 最后怎么算做成了

讲稿提示：

> 从 Coding Agent 往业务系统走，很容易以为：那业务里也加一个 Agent 不就行了？  
> 让它读上下文，想方案，然后执行。  
> 但这里有个坑。  
>   
> 比如一个人说：我的销售怎么上不去？  
> 这句话看起来很简单，其实系统根本没法直接做。  
> 他到底想要什么？是冲量，还是保利润？是先清库存，还是先控制预算？  
> Agent 应该给什么？是看数据，还是给方案，还是直接改规则？  
> 最后怎么算成功？数字涨了但成本也涨了，算不算成功？  
>   
> 所以业务复杂，不是因为它多高级。  
> 而是用户说出来的是一句人话，系统真要做事的时候，必须落到很具体的动作上。  
> 更麻烦的是，很多业务动作一旦执行出去，就不是一句撤销能解决的。  
> 改代码错了可以看 diff、跑测试、revert。  
> 但预算花出去了，库存动了，规则改了，影响可能已经出去了。  
> 所以业务 Agent 不能听懂一句人话之后，就直接去动真实系统。  
> 中间必须有一层。

## 2.3 协同文档前时代

页面要点：

> 协同文档成熟之前，多人一起改文档最痛苦的不是写，而是合并。

典型痛点：

- 附件来回发
- 文件名变成 final、final_v2、final_v7
- 两个人都改同一段，不知道留谁
- 一个人删了段落，另一个人又在里面加内容
- 插图、表格、标题、组件混在一起，人工合并很崩溃

讲稿提示：

> 讲业务 Agent 之前，我想先讲一个大家都熟的东西：协同文档。  
> 现在我们觉得多人一起写文档很正常。  
> 但在协同文档真正成熟之前，一起改文档是很痛苦的。  
> 一个人写完发附件，另一个人改完再发回来。  
> 文件名很快就变成最终版、最终版修改、最终版真的不改了、final_v7。  
>   
> 最痛苦的不是写。  
> 是合并。  
> 两个人都改了同一段怎么办？  
> 一个人删了一个段落，另一个人在这个段落里加了一句话怎么办？  
> 一个人插了表格，另一个人在表格前面加了标题怎么办？  
>   
> 这些组合几乎是没法穷举的。  
> 所以协同文档解决的问题，和业务 Agent 有一点像：人的操作很多，顺序很多，组合很多，不能靠穷举。

## 2.4 协同文档怎么解这个事

页面要点：

> 协同文档不是理解所有人的想法，而是把无限操作先变成有限 action。

以 json0 / OT 这类思路为例，系统内部会把用户操作拆成几类小动作：

- 在某个位置插入文字
- 在某个位置删除文字
- 在列表里插入、删除、移动一个东西
- 改某个对象上的一个属性

讲稿提示：

> 那协同文档是怎么解的？  
> 它不是靠理解所有人到底想怎么写文档。  
> 它做了一件很朴素但很关键的事：先把用户操作变成系统能处理的几种小动作。  
>   
> 用户表面上是在加粗、换行、插图、改组件。  
> 但系统内部会把它们拆成插入、删除、移动、改属性这些 action。  
> 这些 action 是有限的。  
> 每个 action 带着位置、版本、内容。  
> 如果两个人同时改，系统再根据这些信息调整顺序、处理冲突，最后合成一份文档。  
>   
> 所以协同文档真正厉害的地方，不是它知道用户所有想法。  
> 而是它做了一个转换：  
>   
> 人想怎么改都行。  
> 但系统先把它变成几种 action。  
> 然后再合并。  
>   
> 这个思路对业务 Agent 很有启发。  
> 业务里用户也会说很多人话，比如“这个项目推不动”“风险有点大”“销售怎么上不去”。  
> 这些话没法穷举。  
> 但我们也许可以先把它们变成几种业务动作。  
> 我先把这个叫 WGS。  
> 说白了就是：无限意图，有限操作。

## 2.5 Amazon Canvas：把开放意图放进业务画布

页面要点：

> Amazon Seller Canvas 有意思的地方，不是多了一个 AI 页面，而是它先让业务在 Canvas 里摊开。

它做了几件事：

- 把经营数据、趋势、库存、广告、流量这些信息拉到一个地方
- 让 AI 先生成候选建议，而不是直接执行
- 让人继续追问、加条件、看影响
- 让方案先停在 Canvas 里
- 确认之后，再进入真实业务动作

讲稿提示：

> 回到 Amazon Seller Canvas。  
> 它让我觉得有意思的地方，不是它做了一个 AI 页面。  
> 而是它没有让 AI 一上来就动真实业务。  
>   
> 卖家问的问题也很人话。  
> 比如帮我看看销售表现，哪些库存要补，广告怎么调，页面哪里要优化。  
> 这些都不是一个按钮。  
>   
> Amazon 的做法，是先把相关数据和建议放进 Canvas。  
> 销售、库存、广告、流量、历史趋势，先拉到一个地方。  
> 然后让卖家看，让卖家问，让卖家加条件。  
> 比如预算不能涨，只看风险比较高的部分，如果打折会怎样，如果补货会怎样。  
>   
> 它先在 Canvas 里试。  
> 先别真的改价格，先别真的下单，先别真的动广告。  
> 所以我觉得它像一个业务沙盘。  
> 先摊开，先比较，先让人看一眼。  
> 确认以后，再进入真实业务。

## 2.6 Amazon 把人话变成什么动作

页面要点：

> Amazon 也在做“无限意图到有限操作”的事。

可以粗看成四类动作：

- 看数据：先把销售、库存、广告、流量看清楚
- 试方案：补货、折扣、预算、页面优化先在画布里比一比
- 给建议：哪里风险高，哪里值得动，哪里先别动
- 再执行：人确认以后，再写进真实系统

讲稿提示：

> 如果用刚才协同文档那个思路来看，Amazon 也在做一件类似的事。  
> 卖家说的是人话。  
> 但系统不能直接执行一句人话。  
>   
> 所以它先把问题变成几类动作。  
> 第一，看数据。销售、库存、广告、流量先看清楚。  
> 第二，试方案。补货、折扣、预算、页面优化，先在画布里比一比。  
> 第三，给建议。哪里风险高，哪里值得动，哪里先别动。  
> 第四，再执行。卖家点头以后，再进入真实系统。  
>   
> 所以它不是让 Agent 直接接管业务。  
> 它是把一句很空的话，先变成几种系统能理解的动作。  
> 这就是我说的 WGS。  
> 说白了：人话先进草稿区，先变成几种动作，人看完，再真的执行。

## 2.7 从 Canvas 读出 human in the loop

页面要点：

> Human in the Loop 不是每一步都弹确认框，而是真执行前，人能看懂、能改、能说不。

这里能读出几个控制点：

- 先停一下：方案先留在 Canvas
- 人看一眼：能追问、能改条件、能拒绝
- 再执行：确认后才进入真实系统
- 留记录：执行了什么、为什么执行、谁确认过，都要留痕
- 能补救：后面出问题，至少知道从哪里退、找谁负责

讲稿提示：

> 到这里，第二章真正想讲的东西才出来。  
> 我不是为了介绍 Amazon。  
> Amazon 只是一个例子。  
> 我是想借它，把几个经常被说得很玄的词，拉回到产品动作里。  
>   
> 第一，Human in the Loop。  
> 这个词经常被讲得很大。  
> 但放到这里，其实很简单。  
> 不是每一步都弹确认框。  
> 那样很烦，人会变成一个确认按钮。  
> 真正重要的是，在动作真的写进业务系统之前，人能看懂，能改，能说不。  
>   
> 第二，Checkpoint。  
> 也就是先停一下。  
> 方案、预测、清单，先留在 Canvas 里。  
> 不要一上来就改真实系统。  
>   
> 第三，Recovery。  
> 业务不像代码，可以很轻松地 git revert。  
> 所以前面最好能打回去，后面真的执行了，也要有记录、有责任、有补救办法。  
>   
> 这些概念如果说英文，会显得很高级。  
> 但放到产品里，其实就是几句很朴素的话：  
> 先别真动。  
> 先让我看一眼。  
> 不对我能打回。  
> 真执行了也要留痕。

## 2.8 第二章收束

页面要点：

> 不要让 Agent 猜所有业务，先让它变成几种动作。

第二章收束成三句话：

- 业务里的真实场景太多，不能靠穷举
- 协同文档给了一个启发：无限操作可以先变成有限 action
- Amazon Canvas 给了一个实践：开放意图先放进沙盘，人看完再执行

讲稿提示：

> 第二章我想收束成一句话：不要让 Agent 猜所有业务，先让它变成几种动作。  
>   
> 业务里的真实场景太多了。  
> 用户说的话也太散了。  
> 如果我们指望 Agent 把所有业务场景都想明白，再直接执行，这个风险太大。  
>   
> 更现实的办法，是像协同文档那样：先把人的意图变成几种动作。  
> 再像 Amazon Canvas 那样：先放在一个沙盘里摊开。  
> 人看完，能改，能拒绝，能确认。  
> 最后再进入真实系统。  
>   
> 所以 WGS 这个词如果说得土一点，就是：  
> 人话先别直接进系统。  
> 先变成草稿动作。  
> 人看完，再执行。

# 3. 设计实践

> 回到我们自己的项目，看 Agent 怎么落地

## 3.1 回到我们自己的项目

页面要点：

> 前面讲的是思路，这一章要落到我们自己的协同文档项目里。

这里最关心四件事：

- Agent 改了什么，要看得见
- 人能不能看懂这一轮改动
- 接不接受，要由人决定
- 不对的时候，能不能退回去

讲稿提示：

> 前面两章，我先讲了 Coding Agent 为什么先跑起来，也讲了 Amazon 这种业务实践给我的启发。  
> 到第三章，我想回到我们自己的项目。  
>   
> 这里不是再讲一个抽象框架。  
> 我更关心的是，Agent 真的进到协同文档以后，用户会不会慌。  
> 它改了什么？  
> 为什么这么改？  
> 我能不能先看懂？  
> 我不想要这次改动，能不能退回去？  
>   
> 所以第三章就围绕几个很具体的产品动作讲：对比、撤回、记录、决策。

## 3.2 撤回边界：人的编辑和 Agent 编辑不能混

页面要点：

> Ctrl+Z 退人的手动编辑，撤回 Agent 修改退这一轮 AI 改动，两条线不能混。

要讲清楚：

- 人的编辑是一条历史
- Agent 的编辑是另一条历史
- 两边都要能退
- 但不能互相误伤

讲稿提示：

> 在协同文档里，第一个要说清楚的是撤回边界。  
> `Ctrl+Z` 本来是给人的手动编辑用的。  
> Agent 对话里的撤回，应该只针对这一轮 AI 改动。  
>   
> 这两条线要分开记、分开退。  
> 否则用户会很不安。  
> 他按一下撤回，到底退的是自己刚才敲的字，还是退 Agent 改的内容？  
> 如果这个边界不清楚，用户就不敢用。  
>   
> 所以这里不是一个小交互问题。  
> 它其实是在告诉用户：人的操作和 Agent 的操作，系统分得清。  
> 分得清，用户才敢让 Agent 进来。

## 3.3 让用户先看懂：side-by-side diff

页面要点：

> 不要把结果突然写进去，要先让用户看懂这一轮到底改了哪里。

展示上可以做两层：

- 块级对齐：先看哪一段、哪一块发生变化
- 文本级细化：再看具体改了哪些字句

讲稿提示：

> 第二个问题是，Agent 的修改怎么被看见。  
> 如果只是把结果直接写进文档，用户会觉得文档是突然变出来的。  
> 他不知道哪里变了，也不知道这次改动重不重。  
>   
> 所以这里要做 side-by-side diff。  
> 先做块级对齐，让用户知道哪一段发生了变化。  
> 再在局部做文本级细化，让用户看清具体改了哪些字句。  
>   
> 用户真正关心的，不是系统内部跑了多少步。  
> 他关心的是：这一轮到底改了哪里，改得重不重，值不值得接收。  
> 看懂以后，他才有资格做决定。

## 3.4 记录 / 展示 / 决策

页面要点：

> Agent 进协同文档，不是只做一个结果页，而是要把事实、展示、决策分开。

三层拆开：

- 记录层：记住谁改的、改了什么、什么时候改的
- 展示层：把记录翻译成人能看懂的差异
- 决策层：接受、拒绝、撤回、继续修改

讲稿提示：

> 我最后把这套设计收成三层：记录、展示、决策。  
>   
> 底层先记录事实。  
> 谁改的，改了什么，什么时候改的，这些要先记清楚。  
>   
> 中间再做展示。  
> 不能把一堆内部记录丢给用户，要翻译成人能看懂的差异。  
>   
> 最上面才是决策。  
> 用户可以接受，可以拒绝，可以撤回，也可以继续让 Agent 修改。  
>   
> 这一层层分开之后，Agent 才不是一个漂在文档外面的聊天框。  
> 它开始被纳入协同文档自己的工作流里。

## 3.5 第三章收束

页面要点：

> Agent 进入协同文档，关键是透明、可对比、可回滚。

这一章收成三句话：

- 透明：动作要看得见
- 可对比：结果要能被人判断
- 可回滚：不合适要能退

讲稿提示：

> 所以第三章我想讲的，不是追一个热词。  
> 而是回到我们自己的项目，看看 Agent 怎么一步一步进到真实工作流里。  
>   
> 真正重要的，是它的动作能不能被看见，结果能不能被对比，错了能不能撤回。  
> 只有做到这三件事，Agent 才不是一个突然往文档里写东西的黑盒。  
> 它才会变成协同文档里一个可以被理解、可以被判断、可以被控制的参与者。

# 4. 留在台上

> 尽情使用它，探索它的边界

## 4.1 尽情使用它

页面要点：

> 面对 Agent，最危险的不是用错，而是一直站在岸边看。  
> 只有把 token 投到真实任务里，它才会变成文章、代码、方案和原型。

建议表达：

- 尽情使用它
- 拥抱它
- 把它放进真实工作里
- 不要只看别人的案例
- 先让投入变成可见产物，再判断哪些东西值得沉淀

讲稿提示：

> 我越来越觉得，面对 Agent，最危险的不是用错，而是一直站在岸边看。  
> 你不真正把它放进自己的工作，就永远只能听别人讲它强或者不强。  
> 但 Agent 的能力边界，很多时候不是看测评看出来的，是你在真实任务里撞出来的。  
> 今天我们投入的 token，可能先变成文章、代码、方案和原型。  
> 很多代码最后会被丢进垃圾桶，很多想法最后不会上线，但这不完全是浪费。  
> 它至少让想法更快变成了可以看、可以改、可以否定的东西。

## 4.2 从投入到产物，再到成果

页面要点：

> 按投入、产出、成果来看，大部分企业或许还没有真正享受到 AI 红利。

三层判断：

- 投入：模型费用、token、工具、学习成本、流程改造成本
- 产出：文章、代码、方案、原型、报告、Review 结果
- 成果：真实效率提升、收入增长、质量改善、组织能力沉淀

讲稿提示：

> 如果按投入、产出、成果来看，我觉得大部分企业或许还没有真正享受到 AI 红利。  
> 投入已经发生了，token 在消耗，工具在接入，人也在学习。  
> 产出也开始出现了，文章、代码、方案、原型、报告，确实比以前出来得更快。  
> 但这些产出还不等于成果。  
> 成果要看它有没有进入真实流程，有没有被复用，有没有降低成本、提升质量、带来业务结果。  
> 所以短期里，我们会看到大量废稿、大量废代码、大量没有上线的探索。  
> 这很正常。AI 红利不是 token 自动兑换出来的，它需要人把产物筛选、组织、沉淀成真正的工作方法。

## 4.3 不是替换人，而是重组任务

页面要点：

> AI 还没有真正做到一对一替换掉 30% 的员工。  
> 它不是可以直接拔插替换的零件。

建议表达：

- AI 在很多任务上已经比初级白领强
- 但在另一些任务上，它仍然不如人类稳定
- 公司不是一堆可拆卸的人力零件
- 短期红利不是直接替换 10%、20%、30% 的员工
- 更真实的变化，是任务被拆开、产物变多、流程被重组

讲稿提示：

> 我觉得这里也要避免一个误解。  
> AI 并没有真正做到一对一地替换掉 30% 的员工，这点大家应该能达成共识。  
> 它确实在很多任务上比初级白领强，比如写草稿、整理信息、生成代码、做初步分析。  
> 但在另一些任务上，它又不如人类稳定，尤其是涉及复杂语境、组织关系、风险判断和长期责任的事情。  
> 所以它绝对不是一个可以直接拔插替换的零件，更不可能直接取代一家公司 10%、20% 甚至 30% 的人。  
> 更真实的变化是：任务会被重新拆开，人的工作重心会被重新分配，产物会暴增，流程会被重组。  
> AI 红利如果会发生，也不是简单从“减少多少人”开始，而是从“同样的人能不能组织出更多有效成果”开始。

## 4.4 探索它的边界

页面要点：

> 知道边界，内心才不会慌。

要探索的边界：

- 它什么时候很强
- 它什么时候会胡来
- 它需要什么上下文
- 它适合什么粒度的任务
- 它在哪些地方必须让人接手
- 它错了以后你怎么发现、怎么纠正

讲稿提示：

> 很多人面对 Agent 会慌，是因为不知道它的边界。  
> 不知道它什么时候可靠，不知道它什么时候会突然跑偏，也不知道自己应该站在哪个位置。  
> 但只要你真正用过足够多次，你就会慢慢知道：哪些任务可以放手，哪些任务要拆小，哪些地方必须检查，哪些结果只能当草稿。  
> 边界不是恐惧的来源，边界是安全感的来源。

## 4.5 长期会变好，短期会阵痛

页面要点：

> 长期看，Agent 会让很多工作变好。  
> 短期看，它一定会带来阵痛。

阵痛来自：

- 原来的经验会被重新定价
- 原来的流程会被重新拆开
- 原来的岗位边界会变得模糊
- 原来的节奏会被迫加快
- 很多人会经历一段“不知道自己站在哪里”的时期
- 企业会经历一段“投入很多、产物很多、成果还不稳定”的时期
- 管理者会发现“用 AI”不等于“立刻减少人”

讲稿提示：

> 我不想把这件事讲得太轻松。  
> 长期看，它会让很多工作变好；但短期看，它一定会带来阵痛。  
> 因为它不是多了一个工具，而是改变了人的工作重心。  
> 有些经验会被重新定价，有些流程会被重新拆开，有些岗位边界会变得模糊。  
> 对企业来说，也会经历一段很尴尬的时期：投入很多，产物很多，但成果还不稳定。  
> 这段时间不舒服，是正常的。关键不是假装没有阵痛，而是在阵痛里建立新的判断力。

## 4.6 你准备好了吗

页面要点：

> 问题不是 Agent 会不会进入业务系统。  
> 问题是：它进入的时候，你站在哪里？

可以问听众：

- 你有没有把 Agent 放进真实任务里？
- 你知道它在你工作里的边界吗？
- 你知道哪些事可以交给它，哪些事必须你来判断吗？
- 你有没有开始重建自己的工作方法？

讲稿提示：

> 所以最后我想问的不是“你看好 Agent 吗”。  
> 我想问的是：你准备好了吗？  
> 不是准备好被它替代，而是准备好和它一起工作。  
> 准备好把它用到真实任务里，准备好探索它的边界，准备好重新理解自己的价值。

## 4.7 留在台上

页面要点：

> 长期会变好，短期会阵痛。  
> 但真正重要的是：留在台上。

收束表达：

> Agent 不会等我们完全理解它以后再改变世界。  
> 它会一边变强，一边进入我们的工作。  
> 所以最好的姿态不是观望，也不是恐慌，而是上手、试错、校准、站稳。  
> 尽情使用它，拥抱它，也探索它的边界。  
> 你越理解它的边界，越能在变化里保持从容。  
> 留在台上，才有机会参与下一幕。
