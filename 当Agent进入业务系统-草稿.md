# 当 Agent 进入业务系统

> 一份面向 PPT 的内容草稿。  
> 目标不是解释“AI 很强”，而是讲清楚：Agent 如何从代码世界进入真实业务系统，以及业务系统为了接纳 Agent 需要重新设计哪些边界。

## 内容边界

这份 PPT 先聚焦四件事：

- 我与 AI：从模型入口到掌控 Agent
- 控制空间：为什么 Coding Agent 先成立
- 设计实践：Agent 进入协同办公笔记系统
- 留在台上：尽情使用它，探索它的边界

暂时不展开：

- 大模型底层原理
- 各家模型能力对比
- 复杂的多 Agent 架构
- 完整商业化方案

---

## 0. 开场页

### 当 Agent 进入业务系统

副标题可选：

> 从 Coding Agent 到业务协作系统的一次产品设计思考

开场表达：

> 过去我们把 AI 看成一个回答问题的工具。  
> 但 Agent 出现以后，AI 不再只是生成内容，而是开始理解任务、调用工具、参与流程，并对业务状态产生真实影响。  
> 当它进入业务系统，问题就从“它聪不聪明”变成了“它能不能被组织安全、稳定、可控地使用”。

---

# 1. 我与 AI

> 从模型入口到掌控 Agent

## 1.1 为什么是我来讲这个话题

页面要点：

- 我较早开始接触 AI，也长期把 AI 放进真实工作流里使用
- Coding Agent 是我最早深度感受到的 Agent 形态
- 在代码世界里，Agent 已经不只是补全工具，而是可以读代码、改代码、跑测试、解释系统
- 我对 AI 的兴趣不是“工具尝鲜”，而是持续观察它如何改变人的工作方式

讲稿提示：

> 我最早感受到 Agent 不是在办公软件里，而是在代码世界里。  
> Coding Agent 很特殊，因为代码本身有明确的结构、文件、测试、版本控制和回滚机制。  
> 这让 Agent 可以比较早地进入一个真实工作环境。  
> 但也正因为这样，我开始意识到：如果 Agent 要进入更复杂的业务系统，它需要的不只是模型能力，还需要系统边界。

## 1.2 不同阶段的不同局限性

页面要点：

- 第一阶段：AI 像搜索和问答，能回答，但不理解业务现场
- 第二阶段：AI 像助手，能生成内容，但结果需要人复制、粘贴、判断
- 第三阶段：Agent 开始调用工具，但容易变成黑盒执行
- 第四阶段：业务 Agent 需要被嵌入权限、流程、数据、审计和协作关系中

讲稿提示：

> 每个阶段的问题都不一样。  
> 问答阶段的问题是“不知道上下文”。  
> 生成阶段的问题是“结果停留在文本里”。  
> 工具调用阶段的问题是“它真的开始做事了，但我们未必知道它为什么这么做”。  
> 到业务系统阶段，真正的问题变成了：谁授权它做？它能改什么？错了怎么办？过程怎么复盘？

## 1.3 一个核心判断

页面要点：

> 我在代码世界里学到的，不是“Agent 可以替人写代码”。  
> 而是：只要一个系统有清晰的边界、验收点和恢复能力，Agent 就有机会承担真实执行。

引出第二部分：

> 很多程序员其实已经和我一样，全面拥抱 Agent 了。  
> 这不是因为程序员更愿意冒险，而是因为代码世界天然有一套控制系统。  
> 所以下面我们先认识一下：代码世界是怎么接住 Agent 的，它依靠哪些控制系统，又形成了哪些体系语言。

可以强调：

- 聊天框解决的是交互入口
- Agent 解决的是任务执行
- 业务系统关心的是状态变化
- 组织关心的是责任边界
- 第二部分要讲的，就是代码世界为什么先给 Agent 提供了一个稳定工作的工程空间

# 2. 控制空间

> 为什么 Coding Agent 先成立

## 2.1 为什么发展最快的是 Coding Agent

页面要点：

> Coding Agent 的成功，不是模型先征服了代码，而是代码世界先给 Agent 提供了一个安全的工程空间。

代码世界至少具备四个条件：

- 可视化：代码、目录、依赖、接口、提交记录、review 历史都摆在明面上
- 相对封闭：输入通常是仓库、文档、规范，输出通常是代码、测试结果、构建结果
- 可验证：可以用 typecheck、test、build、CI、review 判断结果是否成立
- 可回滚：分支、PR、revert、tag、回退发布这些机制早就成熟

讲稿提示：

> 今天回头看 Agent 产品，会发现一个很明显的现象：发展最快的几乎都是 Coding Agent。  
> 表面上看，这像是因为代码更适合大模型。但这个解释其实遮蔽了重点。  
> 代码世界并不简单，它非常复杂。但它的复杂是一种被工程系统包裹过的复杂。  
> 也正因为这个空间足够可见、相对封闭、可验证、可回滚，Agent 才能在这里快速迭代、快速修正、快速进入真实工作流。

## 2.2 业务世界为什么难很多

页面要点：

> 业务世界通常刚好反过来：信息分散、边界流动、验证模糊、后果外溢。

典型困难：

- 信息散落在聊天、表格、邮件、后台系统、知识库和人脑经验里
- 任务执行到一半经常变形，边界不断扩大
- 验证标准不统一，很多时候只能说“先做了再看”
- 很多动作一旦执行出去，不是简单撤销能解决
- Agent 越强，越可能把错误更快地打进真实业务状态

讲稿提示：

> 改一段代码错了，可以回退。  
> 但改一个价格、投一轮广告、发一批促销、改一版商品详情、改一个库存规则，很多时候就不是一句“撤销”能解决的。  
> 它可能已经影响了转化、预算、库存、用户体验，甚至品牌感知。  
> 所以业务 Agent 落地慢，不是因为业务没有价值，也不是因为模型不够聪明，而是因为业务环境本身还不是一个适合 Agent 稳定工作的环境。

## 2.3 没有控制空间，业务 Agent 会撞上现实

页面要点：

> 业务不是一套可以被完整规划的流程。很多时候，伟大不可被规划，业务本来就是在混乱中前行。

所以这里不要急着画一套完美框架，而是先承认现实：

- 业务目标经常是开放的，不是明确指令
- 业务过程经常会变形，不是固定流程
- 业务判断常常依赖人、经验、时机和语境
- 很多结果无法简单验证，也无法轻松回滚
- 所以业务 Agent 不能照搬 Coding Agent 的工作方式

讲稿提示：

> 这一页我不想讲成“业务一定要被重新规划成四层架构”。那会太工程，也不真实。  
> 很多业务就是在混乱里长出来的，甚至很多伟大的东西本来就不可被预先规划。  
> 问题是：Agent 一旦进入这种环境，它不能像人在现场那样靠经验兜住所有模糊性。  
> 所以业务 Agent 真正需要的，不是一套完美流程，而是至少有一个能让它先推演、先成形、先被人看见的空间。Amazon Seller Canvas 就是一个很好的例子。

## 2.4 业务 Agent 的成熟案例：Amazon Seller Canvas

页面要点：

> 对不可轻易回滚的业务场景，正确顺序不是先执行再观察，而是先推演、再审核、再执行。Amazon Seller Canvas 是一个很好的案例。

它不是一个更大的聊天框，而是一个业务沙盘：

- 把业务数据、洞察、建议动作放进可交互工作区
- 让卖家围绕销售、广告、库存、新品发布做推演
- 让 Agent 先形成候选提案，而不是直接碰真实系统
- 让批准、驳回、修改、对比都有明确载体
- 让业务世界具备类似 branch 的安全空间

外部信号：

- Amazon Seller Central 的 Canvas 把聊天、业务数据、洞察、方案和行动建议放进可交互工作区
- 它有价值的地方不是“漂亮界面”，而是让卖家先探索场景、比较方案、理解影响，再决定是否推进到真实动作

讲稿提示：

> 当一个场景不可轻易回滚时，最好的办法不是事后补救，而是事前推演。  
> Amazon 的 Seller Canvas 很有启发性，它不是单纯给 Agent 加了一个漂亮界面，而是把复杂的电商管理先拉进一个可以推演、可以比较、可以审查的空间里。  
> 对业务 Agent 来说，这类 Canvas / 沙盘不是装饰，而是高风险动作进入真实世界前的缓冲层。

## 2.5 为什么“提升销量”会变成“加商品”

页面要点：

> “提升销量”不是一个动作，而是一组增长杠杆。

可能的增长杠杆：

- 库存：热销商品是否缺货，是否需要补货或调拨
- 营销：广告预算、曝光、转化、促销是否需要调整
- 商品页：listing、图片、标题、内容是否需要优化
- 价格：是否需要折扣、调价、清库存
- 履约：FBA、仓储、配送速度、库存成本是否需要优化
- 选品：是否有相邻品类、变体或新品线可以扩张

讲稿提示：

> 这个地方很容易误读。  
> 如果卖家问“怎么提升销量”，Agent 直接回答“去加商品”，听起来像跑偏。  
> 但在业务里，销量增长不是一个单一动作。它可能来自库存、营销、价格、商品页、履约，也可能来自选品扩张。  
> 所以“加商品”不是跳题，而是销量增长在选品扩张上的一条候选路径。Canvas 的价值，是先把开放目标拆成增长杠杆，再把每条杠杆变成可比较、可推演、可确认的方案。

## 2.6 Amazon 的四类业务场景

页面要点：

Amazon 文章里可以拆成四类场景：

- 业务表现分析：销售、流量、趋势、异常、补库存建议
- 营销表现优化：广告花费、曝光、转化、销售提升、预算约束
- 库存决策：补货、延迟补货、折扣清库存、stockout risk
- 新品发布：历史销售、客户洞察、类目需求、竞争强度、launch checklist

讲稿提示：

> 注意这四类场景的共同点：它们都不是纯内容生成。  
> 它们都连接真实业务状态：广告预算、库存、现金流、类目竞争、商品发布。  
> 所以 Canvas 没有把它们变成一句建议，而是变成可以展开、对比、追问、加约束的方案对象。  
> 这就是业务 Agent 的核心能力：不是给一句建议，而是把开放问题收敛成可执行方案。

## 2.7 从 Amazon 得到的方法论

页面要点：

> 业务 Agent 的核心能力，不是“给一句建议”，而是“把开放问题收敛成可执行方案”。

可迁移的方法论：

- 开放目标先不要直接执行
- 先识别业务杠杆
- 再把方案产物化
- 用指标和约束做推演
- 关键节点保留人类判断
- 通过后再进入分阶段执行

讲稿提示：

> 所以我们学 Amazon，不是学它的界面长什么样。  
> 真正要学的是这条路径：开放问题进入控制空间，变成业务对象；业务对象经过推演和审核，再进入真实执行。  
> 这套思路可以迁移到很多业务系统里。下一章我们再把它放到协同办公笔记系统，看会议、文档、任务和项目如何也被这样组织起来。

## 2.8 这一章的结论

页面要点：

> 真正值得做的，不是把公司里的一切都规划成流程，也不是把一切都 Agent 化。

更现实的做法是：

- 承认业务本来就是开放和混乱的
- 不要求 Agent 一上来直接执行
- 先给关键业务动作一个沙盘
- 让方案先成形、被比较、被人判断
- 再决定哪些动作可以进入真实系统

讲稿提示：

> 所以这里不是要把业务世界改造成代码世界。  
> 很多业务之所以有生命力，正是因为它不是被完整规划出来的。  
> 但 Agent 要进入这样的环境，至少需要一个缓冲层：先把开放问题变成方案对象，再让人判断，再进入真实执行。  
> 这就是 Amazon Canvas 给我的启发。  
> 下一章我们再把这套控制空间放进一个具体场景：协同办公笔记系统应该怎么设计 Agent。

# 3. 设计实践

> Agent 进入协同办公笔记系统

## 3.1 为什么选择协同办公笔记系统

页面要点：

- 笔记系统天然承载大量非结构化信息
- 协同办公场景里有会议、任务、项目、知识库和决策记录
- 用户真正需要的不是“再写一篇笔记”，而是从信息中得到行动
- 这是 Agent 很适合进入的业务入口

讲稿提示：

> 笔记系统看起来是内容工具，但在企业里，它其实是很多业务协作的中间层。  
> 会议纪要、项目背景、需求讨论、客户反馈、决策记录，都可能先出现在笔记里。  
> 所以 Agent 如果只做摘要是不够的，它应该像 Amazon Canvas 那样，把开放问题变成可讨论、可确认、可推进的方案对象。

## 3.2 把 Amazon 模式迁移过来

页面要点：

| Amazon Seller Canvas | 协同办公笔记系统 |
|---|---|
| 提升销量 | 推进项目 / 解决风险 |
| 销售、广告、库存数据 | 会议、文档、任务、评论、项目状态 |
| 增长杠杆 | 决策、任务、负责人、风险、依赖 |
| Canvas 沙盘 | 项目行动画布 |
| launch checklist | 行动清单 / 审批清单 |
| Seller confirmation | 责任人确认 / 项目 owner 确认 |

讲稿提示：

> 在办公系统里，用户经常不会说“请创建 5 个任务并分配给 A、B、C”。  
> 他会说“这个项目推进不动了”“会议之后下一步是什么”“这个需求风险在哪里”。  
> 这些也是开放问题。我们要做的不是让 Agent 直接写任务，而是先把问题放进行动画布里，拆成决策、风险、任务、依赖和责任人。

## 3.3 模块边界设计

页面要点：

系统可以先分成四层：

- 内容层：文档、会议、评论、附件、知识卡片
- 业务层：项目、任务、人员、审批、客户或内部流程
- Agent 层：理解、规划、执行、解释、追踪
- 控制层：权限、确认、版本、日志、回滚

讲稿提示：

> 这里最重要的是边界。  
> 内容层负责保存信息，业务层负责承载组织流程，Agent 层负责把意图转化成动作，控制层负责让动作可控。  
> 如果这些边界混在一起，Agent 很快会变成一个“什么都能碰，但没人知道它碰了什么”的系统风险。

## 3.4 一个典型流程：从会议纪要到行动画布

页面要点：

用户上传或记录会议内容后，Agent 可以：

1. 识别会议主题、参与人和项目背景
2. 提取决策、风险、待办和时间点
3. 生成行动画布：任务草案、负责人、依赖、风险、需要确认的点
4. 让项目 owner 在画布里调整和确认
5. 写入任务系统和项目笔记
6. 记录执行日志，支持打回或撤回

讲稿提示：

> 这个流程里，Agent 不是只总结会议，而是把会议里的信息推进到协作系统中。  
> 但它不是偷偷执行。它先理解，再形成行动画布，再让人确认，最后才写入系统。  
> 这和 Amazon Canvas 的逻辑是一致的：先把开放信息组织成方案对象，再决定要不要进入真实执行。

## 3.5 产品形态：不是一个聊天框，而是一组 Agent 能力

页面要点：

可以设计成几种入口：

- 文档旁边的上下文助手
- 会议结束后的自动整理面板
- 项目空间里的风险和待办发现器
- 知识库里的重复内容合并建议
- 任务系统里的进展追踪助手

讲稿提示：

> 聊天框是最容易想到的入口，但不是唯一入口。  
> 在业务系统里，Agent 更应该出现在用户正在工作的地方。  
> 用户在看文档，它就理解文档；用户在开会，它就整理会议；用户在看项目，它就发现风险和下一步。

## 3.6 状态归属与权限设计

页面要点：

- 用户拥有最终决策权
- 文档系统拥有内容版本
- 任务系统拥有任务状态
- Agent 拥有执行过程和解释记录
- 权限系统决定 Agent 能看什么、能改什么、需要谁确认

讲稿提示：

> Agent 不应该拥有业务状态的最终解释权。  
> 它可以建议、生成、执行被授权的动作，但每一种状态都应该归属到原本负责它的业务模块。  
> 这样系统才不会因为引入 Agent 而变成一团混乱。

# 4. 留在台上

> 尽情使用它，探索它的边界

## 4.1 尽情使用它

页面要点：

> 面对 Agent，最危险的不是用错，而是一直站在岸边看。  
> 只有把 token 投到真实任务里，它才会变成文章、代码、方案和原型。

建议表达：

- 尽情使用它
- 拥抱它
- 把它放进真实工作里
- 不要只看别人的案例
- 先让投入变成可见产物，再判断哪些东西值得沉淀

讲稿提示：

> 我越来越觉得，面对 Agent，最危险的不是用错，而是一直站在岸边看。  
> 你不真正把它放进自己的工作，就永远只能听别人讲它强或者不强。  
> 但 Agent 的能力边界，很多时候不是看测评看出来的，是你在真实任务里撞出来的。  
> 今天我们投入的 token，可能先变成文章、代码、方案和原型。  
> 很多代码最后会被丢进垃圾桶，很多想法最后不会上线，但这不完全是浪费。  
> 它至少让想法更快变成了可以看、可以改、可以否定的东西。

## 4.2 从投入到产物，再到成果

页面要点：

> 按投入、产出、成果来看，大部分企业或许还没有真正享受到 AI 红利。

三层判断：

- 投入：模型费用、token、工具、学习成本、流程改造成本
- 产出：文章、代码、方案、原型、报告、Review 结果
- 成果：真实效率提升、收入增长、质量改善、组织能力沉淀

讲稿提示：

> 如果按投入、产出、成果来看，我觉得大部分企业或许还没有真正享受到 AI 红利。  
> 投入已经发生了，token 在消耗，工具在接入，人也在学习。  
> 产出也开始出现了，文章、代码、方案、原型、报告，确实比以前出来得更快。  
> 但这些产出还不等于成果。  
> 成果要看它有没有进入真实流程，有没有被复用，有没有降低成本、提升质量、带来业务结果。  
> 所以短期里，我们会看到大量废稿、大量废代码、大量没有上线的探索。  
> 这很正常。AI 红利不是 token 自动兑换出来的，它需要人把产物筛选、组织、沉淀成真正的工作方法。

## 4.3 不是替换人，而是重组任务

页面要点：

> AI 还没有真正做到一对一替换掉 30% 的员工。  
> 它不是可以直接拔插替换的零件。

建议表达：

- AI 在很多任务上已经比初级白领强
- 但在另一些任务上，它仍然不如人类稳定
- 公司不是一堆可拆卸的人力零件
- 短期红利不是直接替换 10%、20%、30% 的员工
- 更真实的变化，是任务被拆开、产物变多、流程被重组

讲稿提示：

> 我觉得这里也要避免一个误解。  
> AI 并没有真正做到一对一地替换掉 30% 的员工，这点大家应该能达成共识。  
> 它确实在很多任务上比初级白领强，比如写草稿、整理信息、生成代码、做初步分析。  
> 但在另一些任务上，它又不如人类稳定，尤其是涉及复杂语境、组织关系、风险判断和长期责任的事情。  
> 所以它绝对不是一个可以直接拔插替换的零件，更不可能直接取代一家公司 10%、20% 甚至 30% 的人。  
> 更真实的变化是：任务会被重新拆开，人的工作重心会被重新分配，产物会暴增，流程会被重组。  
> AI 红利如果会发生，也不是简单从“减少多少人”开始，而是从“同样的人能不能组织出更多有效成果”开始。

## 4.4 探索它的边界

页面要点：

> 知道边界，内心才不会慌。

要探索的边界：

- 它什么时候很强
- 它什么时候会胡来
- 它需要什么上下文
- 它适合什么粒度的任务
- 它在哪些地方必须让人接手
- 它错了以后你怎么发现、怎么纠正

讲稿提示：

> 很多人面对 Agent 会慌，是因为不知道它的边界。  
> 不知道它什么时候可靠，不知道它什么时候会突然跑偏，也不知道自己应该站在哪个位置。  
> 但只要你真正用过足够多次，你就会慢慢知道：哪些任务可以放手，哪些任务要拆小，哪些地方必须检查，哪些结果只能当草稿。  
> 边界不是恐惧的来源，边界是安全感的来源。

## 4.5 长期会变好，短期会阵痛

页面要点：

> 长期看，Agent 会让很多工作变好。  
> 短期看，它一定会带来阵痛。

阵痛来自：

- 原来的经验会被重新定价
- 原来的流程会被重新拆开
- 原来的岗位边界会变得模糊
- 原来的节奏会被迫加快
- 很多人会经历一段“不知道自己站在哪里”的时期
- 企业会经历一段“投入很多、产物很多、成果还不稳定”的时期
- 管理者会发现“用 AI”不等于“立刻减少人”

讲稿提示：

> 我不想把这件事讲得太轻松。  
> 长期看，它会让很多工作变好；但短期看，它一定会带来阵痛。  
> 因为它不是多了一个工具，而是改变了人的工作重心。  
> 有些经验会被重新定价，有些流程会被重新拆开，有些岗位边界会变得模糊。  
> 对企业来说，也会经历一段很尴尬的时期：投入很多，产物很多，但成果还不稳定。  
> 这段时间不舒服，是正常的。关键不是假装没有阵痛，而是在阵痛里建立新的判断力。

## 4.6 你准备好了吗

页面要点：

> 问题不是 Agent 会不会进入业务系统。  
> 问题是：它进入的时候，你站在哪里？

可以问听众：

- 你有没有把 Agent 放进真实任务里？
- 你知道它在你工作里的边界吗？
- 你知道哪些事可以交给它，哪些事必须你来判断吗？
- 你有没有开始重建自己的工作方法？

讲稿提示：

> 所以最后我想问的不是“你看好 Agent 吗”。  
> 我想问的是：你准备好了吗？  
> 不是准备好被它替代，而是准备好和它一起工作。  
> 准备好把它用到真实任务里，准备好探索它的边界，准备好重新理解自己的价值。

## 4.7 留在台上

页面要点：

> 长期会变好，短期会阵痛。  
> 但真正重要的是：留在台上。

收束表达：

> Agent 不会等我们完全理解它以后再改变世界。  
> 它会一边变强，一边进入我们的工作。  
> 所以最好的姿态不是观望，也不是恐慌，而是上手、试错、校准、站稳。  
> 尽情使用它，拥抱它，也探索它的边界。  
> 你越理解它的边界，越能在变化里保持从容。  
> 留在台上，才有机会参与下一幕。
