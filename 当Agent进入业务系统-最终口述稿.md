## 开场

【切页：开场页】

过去我们讲 AI，很多时候是在讲一个“回答问题”的工具。  
它可以帮我们写文案、生成代码、总结资料、回答一些问题。

但 Agent 出现以后，事情开始变得不一样了。

Agent 不只是回答，它开始理解任务、调用工具、推进流程，甚至对业务系统里的状态产生真实影响。

所以当 Agent 进入业务系统时，问题就不再只是：

> 它聪不聪明？

而是变成：

> 它能不能被组织安全、稳定、可控地使用？

我更想从自己的经历出发，讲一条线：

> 我为什么先在代码世界里感受到 Agent，  
> 为什么 Coding Agent 发展得最快，  
> 以及业务系统如果要接住 Agent，需要先理解什么。

# 第一章：我与 AI

> 从模型入口到掌控 Agent

## 1.1 我与 AI 的关系变化

【切页：Pinned #5「我和AI」】

【图：我和 AI 四阶段总图】

这一页我想先讲我和 AI 的关系变化。

我抽象成了四个阶段：

> 模型入口，模型拓展，Agent 问世，掌控 Agent。

第一阶段，是模型入口。  
ChatGPT 让模型进入大众视野，API 又让它从一个聊天产品，变成开发者可以接入产品的能力。

第二阶段，是模型拓展。  
那时候 AI 还没有真正变成 Agent，但我们已经在想办法让模型知道更多、用到更多地方。

第三阶段，是 Agent 问世。  
AI 不再只是生成一段内容，而是开始接收任务、分解任务、执行、验证，再继续推进。

第四阶段，是掌控 Agent。  
这时候重点不再是“Agent 能不能做”，而是“我如何让它在不同项目里被我掌控”。

这四个阶段，能看到我和AI的协作，更能看到AI的发展。

## 1.3 模型拓展：总图

【切页：Pinned #3「二阶段：模型拓展」】

【图：模型拓展总图】

随着发展，开源界 有个NextChatWeb将提示词放到了工程上，可以自定义角色

也是这个阶段，有幸和古今中外的角色都聊了一波，很有趣，也有收获

但是，后来发现，可能，单纯聊天没有那么大意思。

所以 如果需要 AI 要进入真实世界解决问题，它需要被拓展

这里的“拓展”至少有几类：

第一类是知识拓展。  
模型自己知道的东西有限，所以我们会用 embedding、向量检索把外部知识接到上下文里。

第二类是行为拓展。  
通过 prompt、结构化输出、微调，让模型输出更稳定，更贴合任务。

第三类是能力拓展。  
比如 function calling / tool calling，让模型连接外部 API、数据和动作，这个时候还没有mcp和skill。

第四类是场景拓展。
把模型放进搜索、客服、代码审核、办公流这些真实场景里。

这一阶段还不是 Agent。  
更准确地说，它是模型从“可调用”走向“可用”的过程。

### 掌控 Agent：社区共识里的四象限

【切页：掌控 Agent 四象限】

【图：社区四象限 / 重要程度 × 执行复杂度】

到了 Codex 这一阶段，我更关心的不是 Agent 能不能做一个巨大任务。

我更关心：

> 我怎么掌控它？

这件事其实不是只有我一个人在想。

社区里已经有很多关于如何使用 Coding Agent 的经验总结。

其中一个很有代表性的表达，就是这张四象限图。

它不是在讨论模型能力，而是在讨论：

> 什么样的任务，适合用什么方式交给 Agent。

纵轴是重要程度。

横轴是执行复杂度。

这两个维度背后，其实是一个共识：

不要只问 Agent 能不能做。

还要问这件事重不重要、复杂不复杂，以及它是否容易被人掌控。

如果重要程度低、执行复杂度也低，它可能只是低优先填充。

如果重要程度高、执行复杂度低，它就是 Quick Win。

如果重要程度低、但执行复杂度高，那就不能直接丢给 Agent，而是要先拆解。

如果重要程度高、执行复杂度也高，那就是重点项目，需要更强的边界和验收机制。

这个四象限对我很有启发。

因为它正好可以引出我自己的几类实践：

- 公司级多 repo，是高重要度、高复杂度
- 开源维护，重要但可以通过 review 机制控制
- 简单交付，复杂度相对低，可以更多看最终结果

也就是说，掌控 Agent 不是一种固定方法。

要先判断这个任务落在哪个象限，再决定是多 Agent 并行、review 回归，还是只验收结果。

下面我拆三个例子讲。

---

## 1.7 掌控 Agent：公司级项目

【切页：Pinned #8「公司级项目」】

【图：公司级多 repo / 多 Agent 并行】

在公司级项目里，重要程度高，执行复杂度也高。

一个项目可能有多个 repo，多个工作区，多个上下文。  
这个时候我可以开多个 Agent 并行推进，但关键不是“让它们狂奔”。

关键是任务颗粒度必须足够小。

每个 Agent 做的事情，都要小到我能验收：

- 我能看懂代码改动
- 我能看页面结果
- 我能知道它有没有越界
- 我能判断要不要继续

所以公司级项目里的掌控方式是：

> 多 Agent 并行，但任务必须小。  
> 并行带来效率，小颗粒度带来掌控。

---

## 1.8 掌控 Agent：开源维护

【切页：Pinned #9「开源维护」】

【图：开源维护 / code review / 架构 review / BDD 回归】

开源维护又是另一种范式。

这种项目不一定每次都要我亲自从头看到尾。  
但只要它是长期维护项目，就必须有质量机制。

我的做法是主动触发几个 skill：

- code review 找具体问题
- 架构 review 看结构是否需要调整
- BDD 行为测试 / dev3000 回归，验证行为有没有坏

这里的掌控感不是来自我盯住每一行代码，而是来自机制。

Agent 完成开发之后，我再让其他 Agent 或 skill 去审它、挑战它、回归它。

所以这一页要表达的是：

> 长期维护项目，掌控来自 review 和回归机制。

---

## 1.9 掌控 Agent：简单交付

【切页：Pinned #7「简单交付」】

【图：简单交付 / spec-kit / 只验收结果】

还有一种场景，是简单交付项目。

比如给别人做一个相对简单的项目，我可以采用 spec-kit 式开发。

这时候我的控制方式会完全不一样。

我不一定看每一行代码。  
我只关心：

- 需求规格有没有写清楚
- 验收标准有没有写清楚
- 最终页面和体验是否满足目标

也就是说，中间过程可以放手，最终只看结果。

所以掌控 Agent 不是一种固定方法。  
不同项目的重要程度、复杂度、维护周期不同，掌控方式也应该不同。

第一章最后我想收束成一句：

> 我在代码世界里学到的，不是 Agent 可以替人写代码。  
> 而是只要一个系统有清晰的边界、验收点和恢复能力，Agent 就有机会承担真实执行。

【切页：转场页】

很多程序员其实已经全面拥抱 Agent 了。  
这不是因为程序员更愿意冒险，而是因为代码世界天然有一套控制系统。

所以下一章，我们从 Coding Agent 讲起，看看为什么它能先跑出来。

# 第二章：控制空间

> 为什么 Coding Agent 先成立

## 2.1 Coding Agent 为什么先成立

【切页：Pinned「第二段：2.1」】

【图：Coding Agent 工作空间】

今天如果回头看 Agent 产品，会发现一个很明显的现象：

> 发展最快的，几乎都是 Coding Agent。

表面上看，这像是因为代码比较适合大模型。

但如果只停留在这个解释上，重点会被遮蔽。

真正的原因不是代码更简单。  
代码世界一点也不简单，它非常复杂。

但代码世界的复杂，是一种被工程系统包裹过的复杂。

它至少有四个特点：

第一，它是可视化的。  
代码、目录、依赖、接口、提交记录、review 历史，都摆在明面上。

第二，它是相对封闭的。  
输入通常是仓库、文档、规范；输出通常是代码、测试结果、构建结果。

第三，它是可验证的。  
我们可以用 typecheck、test、build、CI、review 去判断结果是否成立。

第四，它是可回滚的。  
分支、PR、revert、tag、回退发布，这些机制早就成熟了。

所以 Coding Agent 的成功，不是模型先征服了代码。

而是：

> 代码世界先给 Agent 提供了一个稳定工作的工程空间。

---

## 2.2 业务为什么复杂

【切页：Pinned「第二段：2.2」】

【图：业务世界为什么难】

从 Coding Agent 往业务系统走，第一反应很容易是：

那业务系统是不是也加一个 Agent 就行了？

让它读上下文，想方案，然后去执行。

但这里有个很大的坑。

代码世界虽然复杂，但它有仓库、有文件、有 diff、有测试、有版本控制。
甚至现在讲 SPDD，本质上也是把需求、约束、任务拆解，写成一种可以反复改、可以评审、可以同步给 Agent 的东西。

所以代码很复杂，但它有一套工程架子兜着。

业务世界不一样。

业务不是简单的“流程更多一点”。

它难在很多事情一开始就是空的。

比如一个卖家说：

> 我的销量为什么上不去？

这句话看起来很简单，其实里面有三个问题都没说清楚。

第一，他到底想要什么？

是想冲销量，还是想保利润？
是想清库存，还是先把广告投得更准一点？

第二，Agent 应该给什么？

是建议补货，还是降价，还是加广告，还是改页面？

第三，最后怎么算做成了？

销量涨了，但利润掉了，算成功吗？
广告转化好了，但库存压住了，算成功吗？

所以业务复杂，不是因为它看起来高级。

而是用户说出来的是一句人话，但系统真要做事的时候，必须落到很具体的动作上。

更麻烦的是，很多业务动作一旦执行出去，就不是一句撤销能解决的。

改一段代码错了，可以看 diff、跑测试、revert。
但改一个价格、投一轮广告、发一批促销、改一个库存规则，影响可能已经外溢到预算、库存、客户体验和品牌感知里。

所以业务 Agent 的难点不是“它会不会回答”。

而是：

> 它不能听懂一句人话之后，就直接去动真实系统。

中间必须有一层。

---

## 2.3 协同文档前时代

【切页：协同文档前时代】

【图：附件来回发 / final_v7 / 人工合并】

讲这个之前，我想先说一个大家都熟的场景：协同文档。

现在我们觉得多人一起写文档很正常。

但在协同文档真正成熟之前，一起改文档是很痛苦的。

一个人写完，发一个附件。

另一个人改完，再发回来。

文件名就开始变成：

> 最终版
> 最终版-修改
> 最终版-真的不改了
> final_v7

最痛苦的不是写。

是合并。

两个人都改了同一段，怎么办？

一个人删了一个段落，另一个人在这个段落里加了一句话，怎么办？

一个人插了表格，另一个人在表格前面加了标题，怎么办？

如果全靠人肉合并，就很崩溃。

所以协同文档解决的问题，和业务 Agent 有一点像：

> 人的操作很多，顺序很多，组合很多，不能靠穷举。

那它怎么解决？

不是靠理解所有“人到底想怎么写文档”。

而是把这些动作，先变成系统能处理的几种小动作。

---

## 2.4 协同文档怎么解这个事

【切页：协同文档 / json0】

【图：人想怎么改 → 几种 action → 合并】

以 json0 这类 OT 思路为例。

用户表面上是在加粗、换行、插图片、改组件。

但系统内部，会先把它们变成一些更小的动作：

- 在哪个位置插入一段文字
- 在哪个位置删除一段文字
- 在列表里插入、删除、移动一个东西
- 改某个对象上的一个属性

这些动作是有限的。

然后每个动作带着位置、版本、内容。

如果两个人同时改，系统就根据这些信息去调整顺序，处理冲突，最后合成一份文档。

所以协同文档真正厉害的地方，不是它知道用户所有想法。

而是它做了一个转换：

> 人想怎么改都行。
> 但系统先把它变成几种 action。
> 然后再合并。

这个思路对业务 Agent 很有启发。

业务里面，用户也会说很多人话。

“销量上不去”“项目推不动”“风险很大”。

这些话没法穷举。

但我们也许可以把它们先变成几种业务动作。

我先把这个叫 WGS。

说白了就是：

> 无限意图，有限操作。

---

## 2.5 Amazon Canvas：把问题先摊开

【切页：Amazon 沙盘 / Canvas】

【图：Ask → Canvas → Actions】

回到 Amazon Seller Canvas。

它让我觉得有意思的地方，不是它做了一个 AI 页面。

而是它没有让 AI 一上来就动真实业务。

卖家问的问题也很人话。

比如：

> 帮我看看销售表现。
> 哪些库存要补？
> 促销怎么做得更好？
> 哪些页面要改？

这些都不是一个按钮。

Amazon 的做法，是先把相关数据和建议放进 Canvas。

销售、库存、广告、流量、竞争、历史趋势，先拉到一个地方。

然后让卖家看，让卖家问，让卖家加条件。

比如：

这个月预算不能涨。

只看库存过剩的 SKU。

如果打折，会怎样？

如果补货，会怎样？

它先在 Canvas 里试。

先别真的改价格，先别真的下补货单，先别真的动广告。

所以我觉得它像一个业务沙盘。

先摊开，先比较，先让人看一眼。

确认以后，再进入真实业务。

---

## 2.6 Amazon 把人话变成什么动作

【切页：Amazon 的有限业务 action】

【图：看数据 / 试几种 / 给建议 / 再执行】

如果用刚才协同文档那个思路来看，Amazon 也在做一件类似的事。

卖家说的是人话。

但系统不能直接执行一句人话。

所以它先把问题变成几类动作。

第一，看数据。

销售、库存、广告、流量先看清楚。

第二，试几种。

补货、打折、投广告、改页面，先在画布里比一比。

第三，给建议。

哪里要补货，哪里要调广告，哪里要改页面。

第四，再执行。

卖家点头以后，再去改价格、补货、做 launch checklist。

所以它不是让 Agent 直接接管店铺。

它是把一句很空的话，先变成几种系统能理解的动作。

这就是我说的 WGS。

说白了：

> 人话先进草稿区。
> 先变成几种动作。
> 人看完，再真的执行。

## 2.7 从 Canvas 读出 human in the loop

【切页：从 Canvas 读出控制概念】

【图：先停一下 → 人看一眼 → 再执行】

到这里，第二章真正想讲的东西才出来。

我不是为了介绍 Amazon。

Amazon 只是一个例子。

我是想借它，把几个经常被说得很玄的词，拉回到产品动作里。

第一，Human in the Loop。

这个词经常被讲得很大。

但放到这里，其实很简单。

不是每一步都弹确认框。

那样很烦，人会变成一个确认按钮。

真正重要的是，在动作真的写进业务系统之前，人能看懂，能改，能说不。

第二，Checkpoint。

也就是先停一下。

方案、预测、清单，先留在 Canvas 里。

不要一上来就改真实系统。

第三，Recovery。

业务不像代码，可以很轻松地 git revert。

所以前面最好能打回去，后面真的执行了，也要有记录、有责任、有补救办法。

这些概念如果说英文，会显得很高级。

但放到产品里，其实就是几句很朴素的话：

> 先别真动。
> 先让我看一眼。
> 不对我能打回。
> 真执行了也要留痕。

---

## 2.8 第二章收束

【切页：第二章收束页】

第二章我想收束成一句话：

> 不要让 Agent 猜所有业务，先让它变成几种动作。

业务里的真实场景太多了。

用户说的话也太散了。

如果我们指望 Agent 把所有业务场景都想明白，再直接执行，这个风险太大。

更现实的办法，是像协同文档那样：

先把人的意图变成几种动作。

再像 Amazon Canvas 那样：

先放在一个沙盘里摊开。

人看完，能改，能拒绝，能确认。

最后再进入真实系统。

所以 WGS 这个词如果说得土一点，就是：

> 人话先别直接进系统。
> 先变成草稿动作。
> 人看完，再执行。

【切页：第三章占位】

第三章我会再把这个思路放回到我们自己的场景里：  
如果 Agent 进入协同办公笔记系统，会议、文档、任务和项目，应该怎么被组织成一个可以行动的业务空间。

---

# 第三章：设计实践

> 回到我们自己的项目，看 Agent 怎么落地

## 3.1 先从 why 说起

【切页：回到我们自己的项目】

前面两章，我先讲了 coding agent 为什么先跑起来，也讲了别人的业务实践怎么把 Agent 放进真实流程。到了这里，我想回到我们自己的项目。

这一章按一个更朴素的顺序来讲：`why`、`what`、`how`。先说为什么这里需要 Agent，再说它到底进到哪些环节，最后说我们是怎么把它做成一个能用、能看、能回滚的方案。

在这个项目里，Agent 进入的是协同文档工作流。它会和人的编辑、对比、撤回放在一起，所以我最先关心的，就是它的身份边界和可追踪性。

## 3.2 先准备 prompt 文档

【切页：prompt 文档】

Agent 真正进入系统之前，先要有一份清楚的 prompt 文档。

这份文档要说清楚几件事：这次任务的目标是什么，能改到哪里，不能越到哪里，最后希望交付成什么样。

它看起来像是给 Agent 的说明书，实际上也是给整个协同流程定边界。  
因为只有先把目标、范围和约束写清楚，后面的修改、对比、撤回才有共同依据。

## 3.3 它怎么被看见

【切页：side-by-side diff / 块级 + 文本级】

如果只是把结果直接写进去，用户会觉得文档是突然变出来的。我们希望用户能先看懂，再决定要不要接受。

所以展示上用了并排对比：先做块级对齐，再在局部做文本级细化。

用户真正关心的，不是系统内部做了多少步，而是这一轮到底改了哪里、改得重不重、值不值得接收。

## 3.4 怎么保证能退回去

【切页：Ctrl+Z vs 撤回 Agent 修改】

在协同文档里，`Ctrl+Z` 只应该服务于人的手动编辑，Agent 对话里的撤回则只针对这一轮 AI 的改动。

这两条线要分开记、分开退。这样用户才知道，改动是谁做的，撤回会影响到哪一部分。

这里其实也能看出，Agent 不是一个单独漂在外面的功能点，它是被放进了整个文档编辑系统里，和人的操作一起被管理。

## 3.5 最后收成三层

【切页：记录 / 展示 / 决策】

我最后把这套设计收成三层：

底层记录事实，中层把事实翻译成人能读懂的差异，上层负责接受、拒绝、回滚。

这一层层分开之后，Agent 才能真正进入协同文档，而不只是停留在一个会说话的窗口里。

## 3.6 第三章收束

【切页：透明、可对比、可回滚】

所以第三章我想讲的，不是追一个热词，而是回到我们自己的项目，看看 Agent 怎么一步一步进到真实工作流里。

真正重要的，是它的动作能不能被看见、能不能被理解、能不能被撤回。

---

# 第四章：留在台上

> 尽情使用它，探索它的边界

## 4.1 尽情使用它

【切页：尽情使用它】

最后一章，我想回到人自己。

面对 Agent，我越来越觉得，最危险的不是用错，而是一直站在岸边看。

很多事情你不真正把它放进自己的工作，就永远只能听别人讲它强或者不强。

但 Agent 的能力边界，很多时候不是看测评看出来的，是在真实任务里撞出来的。

今天我们投入的 token，不一定马上变成所谓的业务成果。

它可能先变成文章、代码、方案、原型。

甚至很多代码最后会被丢进垃圾桶，很多想法最后不会上线。

但这不完全是浪费。

因为它至少让想法更快变成了可以看、可以改、可以否定的东西。

原来一个想法可能停留在脑子里，或者停留在一句“有空再说”里。

现在它可以很快变成一版草稿、一段代码、一个页面、一个 demo。

它不一定对，但它出现得更早。

这件事本身就会改变我们的探索速度。

---

## 4.2 从投入到产物，再到成果

【切页：投入 / 产出 / 成果】

如果按投入、产出、成果来看，我觉得大部分企业或许还没有真正享受到 AI 红利。

投入已经发生了。

模型费用在花，token 在消耗，工具在接入，人也在学习。

产出也开始出现了。

文章、代码、方案、报告、原型、Review 结果，确实比以前出来得更快。

但这些产出还不等于成果。

成果要看什么？

要看它有没有进入真实流程。  
有没有被复用。  
有没有降低成本。  
有没有提升质量。  
有没有带来业务结果。  
有没有沉淀成组织能力。

所以短期里，我们会看到大量废稿、大量废代码、大量没有上线的探索。

这很正常。

AI 红利不是 token 自动兑换出来的。

它需要人把这些产物筛选出来，组织起来，沉淀成新的工作方法。

---

## 4.3 它不是拔插替换的零件

【切页：不是替换人，而是重组任务】

这里也要避免一个误解。

AI 并没有真正做到一对一地替换掉 30% 的员工。

这点大家应该能达成共识。

它确实在很多任务上已经比初级白领强。

比如写草稿、整理信息、生成代码、做初步分析。

但在另一些任务上，它又不如人类稳定。

尤其是涉及复杂语境、组织关系、风险判断、长期责任的时候。

所以它绝对不是一个可以直接拔插替换的零件。

更不可能简单地直接取代一家公司 10%、20% 甚至 30% 的人。

更真实的变化是：

任务会被重新拆开。  
人的工作重心会被重新分配。  
产物会暴增。  
流程会被重组。

AI 红利如果会发生，也不是简单从“减少多少人”开始。

而是从“同样的人，能不能组织出更多有效成果”开始。

---

## 4.4 探索它的边界

【切页：知道边界，内心才不会慌】

很多人面对 Agent 会慌，是因为不知道它的边界。

不知道它什么时候可靠。  
不知道它什么时候会突然跑偏。  
也不知道自己应该站在哪个位置。

但只要你真正用过足够多次，你会慢慢知道：

哪些任务可以放手。  
哪些任务必须拆小。  
哪些地方必须检查。  
哪些结果只能当草稿。  
哪些场景必须由人来判断。

所以边界不是恐惧的来源。

边界反而是安全感的来源。

你知道边界，内心才不会慌。

你知道它在哪些地方强，哪些地方弱，你才知道自己应该怎么站稳。

---

## 4.5 长期会变好，短期会阵痛

【切页：长期会变好，短期会阵痛】

我不想把这件事讲得太轻松。

长期看，Agent 会让很多工作变好。

但短期看，它一定会带来阵痛。

因为它不是多了一个工具，而是改变了人的工作重心。

有些经验会被重新定价。  
有些流程会被重新拆开。  
有些岗位边界会变得模糊。  
有些人的节奏会被迫加快。

企业也会经历一段很尴尬的时期：

投入很多，产物很多，但成果还不稳定。

管理者会发现，“用 AI”不等于“立刻减少人”。

个体也会发现，“会用 AI”不等于“可以不判断”。

这段时间不舒服，是正常的。

关键不是假装没有阵痛，而是在阵痛里建立新的判断力。

---

## 4.6 你准备好了吗

【切页：你准备好了吗】

所以最后我想问的不是：

> 你看好 Agent 吗？

我想问的是：

> 你准备好了吗？

不是准备好被它替代，而是准备好和它一起工作。

你有没有把 Agent 放进真实任务里？  
你知道它在你工作里的边界吗？  
你知道哪些事可以交给它，哪些事必须你来判断吗？  
你有没有开始重建自己的工作方法？

Agent 不会等我们完全理解它以后再改变世界。

它会一边变强，一边进入我们的工作。

所以最好的姿态不是观望，也不是恐慌。

而是上手、试错、校准、站稳。

尽情使用它，拥抱它，也探索它的边界。

长期会变好，短期会阵痛。

但真正重要的是：

> 留在台上。

留在台上，才有机会参与下一幕。
