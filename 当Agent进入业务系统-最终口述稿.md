## 开场

【切页：开场页】

过去我们讲 AI，很多时候是在讲一个“回答问题”的工具。  
它可以帮我们写文案、生成代码、总结资料、回答一些问题。

但 Agent 出现以后，事情开始变得不一样了。

Agent 不只是回答，它开始理解任务、调用工具、推进流程，甚至对业务系统里的状态产生真实影响。

所以当 Agent 进入业务系统时，问题就不再只是：

> 它聪不聪明？

而是变成：

> 它能不能被组织安全、稳定、可控地使用？

我更想从自己的经历出发，讲一条线：

> 我为什么先在代码世界里感受到 Agent，  
> 为什么 Coding Agent 发展得最快，  
> 以及业务系统如果要接住 Agent，需要先理解什么。

# 第一章：我与 AI

> 从模型入口到掌控 Agent

## 1.1 我与 AI 的关系变化

【切页：Pinned #5「我和AI」】

【图：我和 AI 四阶段总图】

这一页我想先讲我和 AI 的关系变化。

我抽象成了四个阶段：

> 模型入口，模型拓展，Agent 问世，掌控 Agent。

第一阶段，是模型入口。  
ChatGPT 让模型进入大众视野，API 又让它从一个聊天产品，变成开发者可以接入产品的能力。

第二阶段，是模型拓展。  
那时候 AI 还没有真正变成 Agent，但我们已经在想办法让模型知道更多、用到更多地方。

第三阶段，是 Agent 问世。  
AI 不再只是生成一段内容，而是开始接收任务、分解任务、执行、验证，再继续推进。

第四阶段，是掌控 Agent。  
这时候重点不再是“Agent 能不能做”，而是“我如何让它在不同项目里被我掌控”。

这四个阶段，能看到我和AI的协作，更能看到AI的发展。

## 1.3 模型拓展：总图

【切页：Pinned #3「二阶段：模型拓展」】

【图：模型拓展总图】

随着发展，开源界 有个NextChatWeb将提示词放到了工程上，可以自定义角色

也是这个阶段，有幸和古今中外的角色都聊了一波，很有趣，也有收获

但是，后来发现，可能，单纯聊天没有那么大意思。

所以 如果需要 AI 要进入真实世界解决问题，它需要被拓展

这里的“拓展”至少有几类：

第一类是知识拓展。  
模型自己知道的东西有限，所以我们会用 embedding、向量检索把外部知识接到上下文里。

第二类是行为拓展。  
通过 prompt、结构化输出、微调，让模型输出更稳定，更贴合任务。

第三类是能力拓展。  
比如 function calling / tool calling，让模型连接外部 API、数据和动作，这个时候还没有mcp和skill。

第四类是场景拓展。
把模型放进搜索、客服、代码审核、办公流这些真实场景里。

这一阶段还不是 Agent。  
更准确地说，它是模型从“可调用”走向“可用”的过程。

但这个阶段也还不是 Agent。

因为主流程仍然在人手里，AI 更像是被嵌进流程里的增强能力。  
它能判断、能建议、能辅助 review，但还没有真正承担连续执行。

## 1.7 掌控 Agent：公司级项目

【切页：Pinned #8「公司级项目」】

【图：公司级多 repo / 多 Agent 并行】

到 Codex 这一阶段，我更关心的不是 Agent 能不能做一个巨大任务。

我更关心：

> 我怎么掌控它？

在公司级项目里，重要程度高，执行复杂度也高。

一个项目可能有多个 repo，多个工作区，多个上下文。  
这个时候我可以开多个 Agent 并行推进，但关键不是“让它们狂奔”。

关键是任务颗粒度必须足够小。

每个 Agent 做的事情，都要小到我能验收：

- 我能看懂代码改动
- 我能看页面结果
- 我能知道它有没有越界
- 我能判断要不要继续

所以公司级项目里的掌控方式是：

> 多 Agent 并行，但任务必须小。  
> 并行带来效率，小颗粒度带来掌控。

## 1.8 掌控 Agent：开源维护

【切页：Pinned #9「开源维护」】

【图：开源维护 / code review / 架构 review / BDD 回归】

开源维护又是另一种范式。

这种项目不一定每次都要我亲自从头看到尾。  
但只要它是长期维护项目，就必须有质量机制。

我的做法是主动触发几个 skill：

- code review 找具体问题
- 架构 review 看结构是否需要调整
- BDD 行为测试 / dev3000 回归，验证行为有没有坏

这里的掌控感不是来自我盯住每一行代码，而是来自机制。

Agent 完成开发之后，我再让其他 Agent 或 skill 去审它、挑战它、回归它。

所以这一页要表达的是：

> 长期维护项目，掌控来自 review 和回归机制。

---

## 1.9 掌控 Agent：简单交付

【切页：Pinned #7「简单交付」】

【图：简单交付 / spec-kit / 只验收结果】

还有一种场景，是简单交付项目。

比如给别人做一个相对简单的项目，我可以采用 spec-kit 式开发。

这时候我的控制方式会完全不一样。

我不一定看每一行代码。  
我只关心：

- 需求规格有没有写清楚
- 验收标准有没有写清楚
- 最终页面和体验是否满足目标

也就是说，中间过程可以放手，最终只看结果。

所以掌控 Agent 不是一种固定方法。  
不同项目的重要程度、复杂度、维护周期不同，掌控方式也应该不同。

第一章最后我想收束成一句：

> 我在代码世界里学到的，不是 Agent 可以替人写代码。  
> 而是只要一个系统有清晰的边界、验收点和恢复能力，Agent 就有机会承担真实执行。

【切页：转场页】

很多程序员其实已经全面拥抱 Agent 了。  
这不是因为程序员更愿意冒险，而是因为代码世界天然有一套控制系统。

所以下一章，我们从 Coding Agent 讲起，看看为什么它能先跑出来。

# 第二章：控制空间

> 为什么 Coding Agent 先成立

## 2.1 Coding Agent 为什么先成立

【切页：Pinned「第二段：2.1」】

【图：Coding Agent 工作空间】

今天如果回头看 Agent 产品，会发现一个很明显的现象：

> 发展最快的，几乎都是 Coding Agent。

表面上看，这像是因为代码比较适合大模型。

但如果只停留在这个解释上，重点会被遮蔽。

真正的原因不是代码更简单。  
代码世界一点也不简单，它非常复杂。

但代码世界的复杂，是一种被工程系统包裹过的复杂。

它至少有四个特点：

第一，它是可视化的。  
代码、目录、依赖、接口、提交记录、review 历史，都摆在明面上。

第二，它是相对封闭的。  
输入通常是仓库、文档、规范；输出通常是代码、测试结果、构建结果。

第三，它是可验证的。  
我们可以用 typecheck、test、build、CI、review 去判断结果是否成立。

第四，它是可回滚的。  
分支、PR、revert、tag、回退发布，这些机制早就成熟了。

所以 Coding Agent 的成功，不是模型先征服了代码。

而是：

> 代码世界先给 Agent 提供了一个稳定工作的工程空间。

---

## 2.2 业务为什么混乱

【切页：Pinned「第二段：2.2」】

【图：业务世界为什么难】

业务世界为什么难很多？

因为它通常刚好反过来。

业务信息不是乖乖放在一个仓库里。  
它散落在聊天、表格、邮件、后台系统、知识库，还有人的经验里。

业务目标也经常不是明确指令。  
很多时候用户不会说“请执行 A、B、C 三个步骤”，而是说：

> 销量为什么上不去？  
> 这个项目怎么推进？  
> 这个风险怎么处理？

业务过程也不是固定流程。  
它经常执行到一半就变形，边界不断扩大，判断依赖人、时机和语境。

更麻烦的是，很多业务动作一旦执行出去，就不是一句撤销能解决的。

改一段代码错了，可以回退。  
但改一个价格、投一轮广告、发一批促销、改一版商品详情、改一个库存规则，后果可能已经外溢到预算、库存、用户体验和品牌感知里。

所以业务 Agent 落地慢，不是因为业务没有价值，也不是模型不够聪明。

而是业务世界本身不是一个天然适合 Agent 稳定工作的环境。

这里我也不想把业务讲成一套可以被完整规划的流程。

很多业务本来就是在混乱中前行。  
甚至很多伟大的东西，本来就不可被预先规划。

问题不是要消灭这种混乱，而是：

> Agent 进入这种环境时，至少要有一个让它先推演、先成形、先被人看见的空间。

---

## 2.3 Amazon Canvas：业务沙盘

【切页：Amazon 沙盘 / Canvas】

【图：真实业务问题 → Canvas → 真实系统】

所以我们来看 Amazon Seller Canvas。

我觉得这个案例最有价值的地方，不是它做了一个漂亮界面。

它真正有价值的地方，是把复杂的电商管理，先放进了一个可推演、可比较、可审核的沙盘空间里。

它不是一个更大的聊天框。

聊天框只是在问答。  
Canvas 做的是把问题、数据、洞察、方案、风险、下一步动作组织到同一个空间里。

对不可轻易回滚的业务场景，正确顺序不应该是：

> 先让 Agent 执行，再观察结果。

而应该是：

> 先推演，再审核，再执行。

在 Canvas 里，Agent 先形成候选提案，而不是直接碰真实系统。

人可以比较、追问、修改约束、确认或拒绝。

这就是一个典型的业务沙盘。

它给业务世界提供了一种类似代码 branch 的能力：

> 先在安全空间里形成一个方案版本，再决定是否进入真实系统。

## 2.4 为什么“提升销量”会变成“加商品”

【切页：Amazon 增长杠杆拆解】

【图：提升销量 → 库存 / 营销 / 商品页 / 价格 / 履约 / 选品】

这里有一个很容易误读的问题：

> 原来的问题是“怎么提升销量”，为什么后面会变成“加商品”？

如果把 Agent 理解成聊天机器人，这确实像跑偏。

但在业务里，“提升销量”不是一个动作。  
它是一组增长杠杆。

销量可能来自库存问题。  
热销商品是不是缺货？要不要补货？

销量可能来自营销问题。  
广告预算、曝光、转化、促销策略是不是要调整？

销量可能来自商品页问题。  
listing、图片、标题、内容是不是影响转化？

销量可能来自价格问题。  
要不要折扣、调价、清库存？

也可能来自履约问题。  
配送速度、仓储成本、FBA 策略是否拖累了增长？

当然，也可能来自选品扩张。  
是不是有相邻品类、变体、新品线可以扩张？

所以“加商品 / 新品发布”不是跳题。

它只是“提升销量”这个开放目标，在选品扩张这条增长杠杆上的一个候选路径。

Canvas 的价值就在这里：

> 它不是直接给一句建议，  
> 而是先把开放目标拆成增长杠杆，再把每条杠杆变成可比较、可推演、可确认的方案。

## 2.5 Amazon 的四类业务场景

【切页：Amazon Canvas 方案对象】

【图：开放问题 → 数据汇聚 → 机会识别 → 候选方案 → 沙盘推演 → 分阶段执行】

Amazon 文章里其实给了几类场景。

第一类是业务表现分析。  
比如卖家问：我的产品表现怎么样？销售表现怎么样？

Canvas 不是简单回答一句，而是把销售、流量、趋势、异常、补库存建议放到同一个视图里。

第二类是营销表现优化。  
比如促销活动怎么优化？广告花费、曝光、转化、销售提升之间是什么关系？

卖家还可以加约束：这个月要降低花费，只关注库存过剩商品。  
Canvas 会根据约束重新调整方案。

第三类是库存决策。  
比如哪些产品应该补货？

它不会只给一个补货清单，而是提供多个选项：

- 现在补货
- 延迟补货，继续观察
- 给库存过剩商品打折

并展示这些选择对收入、现金流、缺货风险、仓储费用、竞争位置的影响。

第四类是新品发布。  
比如下一步应该 launch 什么产品？

Canvas 会结合历史销售趋势、客户洞察、类目需求信号、竞争强度，给出扩张路径。

注意这几类场景的共同点：

> 它们都不是纯内容生成。  
> 它们都连接真实业务状态。

所以 Canvas 没有把问题变成一句建议，而是变成一个方案对象。

这个对象可以被展开、对比、追问、加约束、确认，再进入执行准备。

## 2.6 从案例里读出控制概念

【切页：从 Canvas 读出控制概念】

【图：方案对象 → Human in the Loop / Checkpoint / Rollback】

到这里，我们就可以把一些常见术语放回来了。

但我不想一开始就解释术语。  
因为那样会很抽象。

我们可以从 Amazon Canvas 的产品动作里，把这些概念读出来。

第一，Human in the Loop。

在 Canvas 里，人没有被绕过去。  
卖家仍然在关键节点上比较方案、加约束、确认或拒绝。

所以 Human in the Loop 不是每一步都弹确认框。  
而是在关键业务判断上，人仍然保留决策位置。

第二，Checkpoint。

Canvas 里的方案、预测、清单、待确认动作，其实就是进入真实系统前的状态锚点。

它让“Agent 准备做什么”这件事先变成一个可保存、可讨论、可修改的对象。

第三，Rollback 或 Recovery。

业务里的回滚不一定是撤销上一步。

很多业务动作一旦发生，没法真正回到过去。  
所以更重要的是事前推演、延迟进入真实系统、未通过就返回修改、失败后进入补偿。

所以这些术语不是凭空来的。

它们其实都是业务 Agent 想进入真实系统时，必须从产品动作里长出来的控制机制。

## 2.7 第二章收束

【切页：第二章收束页】

第二章我想收束成一句话：

> 真正值得做的，不是把公司里的一切都规划成流程，也不是把一切都 Agent 化。

更现实的做法是：

先承认业务本来就是开放和混乱的。  
不要要求 Agent 一上来直接执行。  
先给关键业务动作一个沙盘。  
让方案先成形、被比较、被人判断。  
再决定哪些动作可以进入真实系统。

所以这里不是要把业务世界改造成代码世界。

很多业务之所以有生命力，正是因为它不是被完整规划出来的。

但 Agent 要进入这样的环境，至少需要一个缓冲层：

> 先把开放问题变成方案对象，  
> 再让人判断，  
> 再进入真实执行。

这就是 Amazon Canvas 给我的启发。

【切页：第三章占位】

第三章我会再把这个思路放回到我们自己的场景里：  
如果 Agent 进入协同办公笔记系统，会议、文档、任务和项目，应该怎么被组织成一个可以行动的业务空间。

## 4.1 尽情使用它

【切页：尽情使用它】

最后一章，我想回到人自己。

面对 Agent，我越来越觉得，最危险的不是用错，而是一直站在岸边看。

很多事情你不真正把它放进自己的工作，就永远只能听别人讲它强或者不强。

但 Agent 的能力边界，很多时候不是看测评看出来的，是在真实任务里撞出来的。

今天我们投入的 token，不一定马上变成所谓的业务成果。

它可能先变成文章、代码、方案、原型。

甚至很多代码最后会被丢进垃圾桶，很多想法最后不会上线。

但这不完全是浪费。

因为它至少让想法更快变成了可以看、可以改、可以否定的东西。

原来一个想法可能停留在脑子里，或者停留在一句“有空再说”里。

现在它可以很快变成一版草稿、一段代码、一个页面、一个 demo。

它不一定对，但它出现得更早。

这件事本身就会改变我们的探索速度。

---

## 4.2 从投入到产物，再到成果

【切页：投入 / 产出 / 成果】

如果按投入、产出、成果来看，我觉得大部分企业或许还没有真正享受到 AI 红利。

投入已经发生了。

模型费用在花，token 在消耗，工具在接入，人也在学习。

产出也开始出现了。

文章、代码、方案、报告、原型、Review 结果，确实比以前出来得更快。

但这些产出还不等于成果。

成果要看什么？

要看它有没有进入真实流程。  
有没有被复用。  
有没有降低成本。  
有没有提升质量。  
有没有带来业务结果。  
有没有沉淀成组织能力。

所以短期里，我们会看到大量废稿、大量废代码、大量没有上线的探索。

这很正常。

AI 红利不是 token 自动兑换出来的。

它需要人把这些产物筛选出来，组织起来，沉淀成新的工作方法。

---

## 4.3 它不是拔插替换的零件

【切页：不是替换人，而是重组任务】

这里也要避免一个误解。

AI 并没有真正做到一对一地替换掉 30% 的员工。

这点大家应该能达成共识。

它确实在很多任务上已经比初级白领强。

比如写草稿、整理信息、生成代码、做初步分析。

但在另一些任务上，它又不如人类稳定。

尤其是涉及复杂语境、组织关系、风险判断、长期责任的时候。

所以它绝对不是一个可以直接拔插替换的零件。

更不可能简单地直接取代一家公司 10%、20% 甚至 30% 的人。

更真实的变化是：

任务会被重新拆开。  
人的工作重心会被重新分配。  
产物会暴增。  
流程会被重组。

AI 红利如果会发生，也不是简单从“减少多少人”开始。

而是从“同样的人，能不能组织出更多有效成果”开始。

---

## 4.4 探索它的边界

【切页：知道边界，内心才不会慌】

很多人面对 Agent 会慌，是因为不知道它的边界。

不知道它什么时候可靠。  
不知道它什么时候会突然跑偏。  
也不知道自己应该站在哪个位置。

但只要你真正用过足够多次，你会慢慢知道：

哪些任务可以放手。  
哪些任务必须拆小。  
哪些地方必须检查。  
哪些结果只能当草稿。  
哪些场景必须由人来判断。

所以边界不是恐惧的来源。

边界反而是安全感的来源。

你知道边界，内心才不会慌。

你知道它在哪些地方强，哪些地方弱，你才知道自己应该怎么站稳。

---

## 4.5 长期会变好，短期会阵痛

【切页：长期会变好，短期会阵痛】

我不想把这件事讲得太轻松。

长期看，Agent 会让很多工作变好。

但短期看，它一定会带来阵痛。

因为它不是多了一个工具，而是改变了人的工作重心。

有些经验会被重新定价。  
有些流程会被重新拆开。  
有些岗位边界会变得模糊。  
有些人的节奏会被迫加快。

企业也会经历一段很尴尬的时期：

投入很多，产物很多，但成果还不稳定。

管理者会发现，“用 AI”不等于“立刻减少人”。

个体也会发现，“会用 AI”不等于“可以不判断”。

这段时间不舒服，是正常的。

关键不是假装没有阵痛，而是在阵痛里建立新的判断力。

---

## 4.6 你准备好了吗

【切页：你准备好了吗】

所以最后我想问的不是：

> 你看好 Agent 吗？

我想问的是：

> 你准备好了吗？

不是准备好被它替代，而是准备好和它一起工作。

你有没有把 Agent 放进真实任务里？  
你知道它在你工作里的边界吗？  
你知道哪些事可以交给它，哪些事必须你来判断吗？  
你有没有开始重建自己的工作方法？

Agent 不会等我们完全理解它以后再改变世界。

它会一边变强，一边进入我们的工作。

所以最好的姿态不是观望，也不是恐慌。

而是上手、试错、校准、站稳。

尽情使用它，拥抱它，也探索它的边界。

长期会变好，短期会阵痛。

但真正重要的是：

> 留在台上。

留在台上，才有机会参与下一幕。
