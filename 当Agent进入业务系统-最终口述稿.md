## 开场

【切页：开场页】

过去我们讲 AI，很多时候是在讲一个“回答问题”的工具。  
它可以帮我们写文案、生成代码、总结资料、回答一些问题。

但 Agent 出现以后，事情开始变得不一样了。

Agent 不只是回答，它开始理解任务、调用工具、推进流程，甚至对业务系统里的状态产生真实影响。

所以当 Agent 进入业务系统时，问题就不再只是：

> 它聪不聪明？

而是变成：

> 它能不能被组织安全、稳定、可控地使用？

我更想从自己的经历出发，讲一条线：

> 我为什么先在代码世界里感受到 Agent，  
> 为什么 Coding Agent 发展得最快，  
> 以及业务系统如果要接住 Agent，需要先理解什么。

# 第一章：我与 AI

> 从模型入口到掌控 Agent

## 1.1 我与 AI 的关系变化

【切页：Pinned #5「我和AI」】

【图：我和 AI 四阶段总图】

这一页我想先讲我和 AI 的关系变化。

我抽象成了四个阶段：

> 模型入口，模型拓展，Agent 问世，掌控 Agent。

第一阶段，是模型入口。  
ChatGPT 让模型进入大众视野，API 又让它从一个聊天产品，变成开发者可以接入产品的能力。

第二阶段，是模型拓展。  
那时候 AI 还没有真正变成 Agent，但我们已经在想办法让模型知道更多、用到更多地方。

第三阶段，是 Agent 问世。  
AI 不再只是生成一段内容，而是开始接收任务、分解任务、执行、验证，再继续推进。

第四阶段，是掌控 Agent。  
这时候重点不再是“Agent 能不能做”，而是“我如何让它在不同项目里被我掌控”。

这四个阶段，能看到我和AI的协作，更能看到AI的发展。

## 1.3 模型拓展：总图

【切页：Pinned #3「二阶段：模型拓展」】

【图：模型拓展总图】

随着发展，开源界 有个NextChatWeb将提示词放到了工程上，可以自定义角色

也是这个阶段，有幸和古今中外的角色都聊了一波，很有趣，也有收获

但是，后来发现，可能，单纯聊天没有那么大意思。

所以 如果需要 AI 要进入真实世界解决问题，它需要被拓展

这里的“拓展”至少有几类：

第一类是知识拓展。  
模型自己知道的东西有限，所以我们会用 embedding、向量检索把外部知识接到上下文里。

第二类是行为拓展。  
通过 prompt、结构化输出、微调，让模型输出更稳定，更贴合任务。

第三类是能力拓展。  
比如 function calling / tool calling，让模型连接外部 API、数据和动作，这个时候还没有mcp和skill。

第四类是场景拓展。
把模型放进搜索、客服、代码审核、办公流这些真实场景里。

这一阶段还不是 Agent。  
更准确地说，它是模型从“可调用”走向“可用”的过程。

### 掌控 Agent：社区共识里的四象限

【切页：掌控 Agent 四象限】

【图：社区四象限 / 重要程度 × 执行复杂度】

到了 Codex 这一阶段，我更关心的不是 Agent 能不能做一个巨大任务。

我更关心：

> 我怎么掌控它？

这件事其实不是只有我一个人在想。

社区里已经有很多关于如何使用 Coding Agent 的经验总结。

其中一个很有代表性的表达，就是这张四象限图。

它不是在讨论模型能力，而是在讨论：

> 什么样的任务，适合用什么方式交给 Agent。

纵轴是重要程度。

横轴是执行复杂度。

这两个维度背后，其实是一个共识：

不要只问 Agent 能不能做。

还要问这件事重不重要、复杂不复杂，以及它是否容易被人掌控。

如果重要程度低、执行复杂度也低，它可能只是低优先填充。

如果重要程度高、执行复杂度低，它就是 Quick Win。

如果重要程度低、但执行复杂度高，那就不能直接丢给 Agent，而是要先拆解。

如果重要程度高、执行复杂度也高，那就是重点项目，需要更强的边界和验收机制。

这个四象限对我很有启发。

因为它正好可以引出我自己的几类实践：

- 公司级多 repo，是高重要度、高复杂度
- 开源维护，重要但可以通过 review 机制控制
- 简单交付，复杂度相对低，可以更多看最终结果

也就是说，掌控 Agent 不是一种固定方法。

要先判断这个任务落在哪个象限，再决定是多 Agent 并行、review 回归，还是只验收结果。

下面我拆三个例子讲。

---

## 1.7 掌控 Agent：公司级项目

【切页：Pinned #8「公司级项目」】

【图：公司级多 repo / 多 Agent 并行】

在公司级项目里，重要程度高，执行复杂度也高。

一个项目可能有多个 repo，多个工作区，多个上下文。  
这个时候我可以开多个 Agent 并行推进，但关键不是“让它们狂奔”。

关键是任务颗粒度必须足够小。

每个 Agent 做的事情，都要小到我能验收：

- 我能看懂代码改动
- 我能看页面结果
- 我能知道它有没有越界
- 我能判断要不要继续

所以公司级项目里的掌控方式是：

> 多 Agent 并行，但任务必须小。  
> 并行带来效率，小颗粒度带来掌控。

---

## 1.8 掌控 Agent：开源维护

【切页：Pinned #9「开源维护」】

【图：开源维护 / code review / 架构 review / BDD 回归】

开源维护又是另一种范式。

这种项目不一定每次都要我亲自从头看到尾。  
但只要它是长期维护项目，就必须有质量机制。

我的做法是主动触发几个 skill：

- code review 找具体问题
- 架构 review 看结构是否需要调整
- BDD 行为测试 / dev3000 回归，验证行为有没有坏

这里的掌控感不是来自我盯住每一行代码，而是来自机制。

Agent 完成开发之后，我再让其他 Agent 或 skill 去审它、挑战它、回归它。

所以这一页要表达的是：

> 长期维护项目，掌控来自 review 和回归机制。

---

## 1.9 掌控 Agent：简单交付

【切页：Pinned #7「简单交付」】

【图：简单交付 / spec-kit / 只验收结果】

还有一种场景，是简单交付项目。

比如给别人做一个相对简单的项目，我可以采用 spec-kit 式开发。

这时候我的控制方式会完全不一样。

我不一定看每一行代码。  
我只关心：

- 需求规格有没有写清楚
- 验收标准有没有写清楚
- 最终页面和体验是否满足目标

也就是说，中间过程可以放手，最终只看结果。

所以掌控 Agent 不是一种固定方法。  
不同项目的重要程度、复杂度、维护周期不同，掌控方式也应该不同。

第一章最后我想收束成一句：

> 我在代码世界里学到的，不是 Agent 可以替人写代码。  
> 而是只要一个系统有清晰的边界、验收点和恢复能力，Agent 就有机会承担真实执行。

【切页：转场页】

很多程序员其实已经全面拥抱 Agent 了。  
这不是因为程序员更愿意冒险，而是因为代码世界天然有一套控制系统。

所以下一章，我们从 Coding Agent 讲起，看看为什么它能先跑出来。

# 第二章：控制空间

> 为什么 Coding Agent 先成立

## 2.1 Coding Agent 为什么先成立

【切页：Pinned「第二段：2.1」】

【图：Coding Agent 工作空间】

今天如果回头看 Agent 产品，会发现一个很明显的现象：

> 发展最快的，几乎都是 Coding Agent。

表面上看，这像是因为代码比较适合大模型。

但如果只停留在这个解释上，重点会被遮蔽。

真正的原因不是代码更简单。  
代码世界一点也不简单，它非常复杂。

但代码世界的复杂，是一种被工程系统包裹过的复杂。

它至少有四个特点：

第一，它是可视化的。  
代码、目录、依赖、接口、提交记录、review 历史，都摆在明面上。

第二，它是相对封闭的。  
输入通常是仓库、文档、规范；输出通常是代码、测试结果、构建结果。

第三，它是可验证的。  
我们可以用 typecheck、test、build、CI、review 去判断结果是否成立。

第四，它是可回滚的。  
分支、PR、revert、tag、回退发布，这些机制早就成熟了。

所以 Coding Agent 的成功，不是模型先征服了代码。

而是：

> 代码世界先给 Agent 提供了一个稳定工作的工程空间。

---

## 2.2 业务为什么混乱

【切页：Pinned「第二段：2.2」】

【图：业务世界为什么难】

业务世界为什么难很多？

因为它通常刚好反过来。

业务信息不是乖乖放在一个仓库里。  
它散落在聊天、表格、邮件、后台系统、知识库，还有人的经验里。

业务目标也经常不是明确指令。  
很多时候用户不会说“请执行 A、B、C 三个步骤”，而是说：

> 销量为什么上不去？  
> 这个项目怎么推进？  
> 这个风险怎么处理？

业务过程也不是固定流程。  
它经常执行到一半就变形，边界不断扩大，判断依赖人、时机和语境。

更麻烦的是，很多业务动作一旦执行出去，就不是一句撤销能解决的。

改一段代码错了，可以回退。  
但改一个价格、投一轮广告、发一批促销、改一版商品详情、改一个库存规则，后果可能已经外溢到预算、库存、用户体验和品牌感知里。

所以业务 Agent 落地慢，不是因为业务没有价值，也不是模型不够聪明。

而是业务世界本身不是一个天然适合 Agent 稳定工作的环境。

问题不是要消灭这种混乱，而是：

> Agent 进入这种环境时，至少要有一个让它先推演、先成形、先被人看见的空间。

---

## 2.3 电商卖家的问题

【切页：电商卖家的问题】

【图：销量为什么上不去 / 增长杠杆】

如果这里直接说“我们来看 Amazon Canvas”，会有点干。

所以先换成一个很具体的问题。

一个电商卖家可能会问：

> 我的销量为什么上不去？

这个问题太常见了。

但它麻烦就麻烦在，它不是一个动作。

不是说你按一下哪个按钮，销量就上去了。

它可能是在问库存：热销品是不是快断货了？

也可能是在问广告：钱花出去了，到底有没有转化？

也可能是在问商品页：图片、标题、listing 是不是没人想点？

也可能是在问价格：是不是该打折，还是打折会不会伤利润？

甚至也可能是在问选品：是不是这个品类已经到头了，要不要上新品？

所以一个很朴素的问题，背后其实是一团业务判断。

而且这些判断不是轻飘飘的。

- 广告会消耗预算
- 补货会占用现金流
- 打折会影响利润和品牌感知
- 新品会带来库存和运营风险

这就是业务和代码不太一样的地方。

代码里，Agent 改错了，我们可以看 diff、跑测试、回滚。

但业务里，它如果直接帮你改广告、调库存、动价格，有些影响就已经出去了。

所以这个问题不能直接变成：

> Agent，帮我提升销量。

至少不能一上来就让它进真实系统。

中间要有一个地方，让它先把问题摊开，把方案摆出来，让人先看一眼。

---

## 2.4 Amazon Canvas：业务沙盘

【切页：Amazon 沙盘 / Canvas】

【图：真实业务问题 → Canvas → 真实系统】

Amazon Seller Canvas 这个案例，有意思的地方就在这里。

它不是说又做了一个聊天框，让卖家和 AI 聊天。

它更像是在卖家的问题和真实系统之间，加了一个沙盘。

卖家问“销量为什么上不去”。

Canvas 不应该只回一句：

> 你去补货。

或者：

> 你去加广告。

因为这句话太快了，也太危险了。

它更像是先把销售、广告、库存、竞争、利润这些东西拉到一个地方。

然后让你看到几种可能：

- 可以补货
- 可以促销
- 可以调整 listing
- 可以控制广告花费
- 也可以考虑新品或相邻品类

关键是，这些东西先停在 Canvas 里。

它没有直接改真实系统。

卖家可以看，可以追问，可以加约束。

比如这个月预算不能再涨，只看库存过剩的商品。

也可以确认，也可以拒绝。

所以我觉得它像一个业务沙盘。

代码里我们可以开 branch。

业务里不能随便开 branch，但 Canvas 至少给了一个类似的缓冲：

> 先有一个方案版本，先别真的动系统。

---

## 2.5 Amazon Canvas：方案对象

【切页：Amazon Canvas 方案对象】

【图：开放问题 → 数据汇聚 → 机会识别 → 候选方案 → 沙盘推演 → 分阶段执行】

所以 Canvas 最重要的，不是回答得多聪明。

而是它把一句很散的问题，变成了一个东西。

我暂时把它叫做方案对象。

说对象好像有点产品经理，但意思很简单：

这不是一句话，是一坨可以被看见、被修改、被确认的东西。

里面会有目标。

卖家到底是在看销售，还是营销，还是库存，还是新品。

里面会有数据。

销售、广告、库存、竞争这些东西，不用全靠人脑子里到处拼。

里面会有几个方案。

不是只有一个答案，可以放在一起比。

里面也会有影响和风险。

收入、现金流、仓储、缺货、成本、用户体验，都会被带出来。

最后还有下一步。

哪些可以分阶段做，哪些要人确认，哪些要打回去再改。

到这里，Canvas 就不再是聊天窗口。

聊天很多时候是说完就过去了。

但方案对象可以停在那，可以展开，可以改，可以被确认。

这对业务系统很重要。

---

## 2.6 从 Canvas 读出控制概念

【切页：从 Canvas 读出控制概念】

【图：2.6-从Canvas读出控制概念 / 方案对象 → Human in the Loop / Checkpoint / Rollback】

到这里，第二章真正想讲的东西才出来。

我不是为了介绍 Amazon。

Amazon 只是一个例子。

我是想借它，把几个经常被说得很玄的词，拉回到产品动作里。

第一，Human in the Loop。

这个词经常被讲得很大。

放到 Canvas 里，其实很简单：

人没有被绕过去。

卖家仍然在关键节点上比较方案、加约束、确认或拒绝。

所以 Human in the Loop 不是每一步都弹确认框，那样也会烦死。

而是在关键业务判断上，人仍然保留决策位置。

第二，Checkpoint。

Canvas 里的方案、预测、清单、待确认动作，其实就是一个停顿点。

Agent 准备做什么，先别急着做。

先变成一个可以保存、可以讨论、可以修改的东西。

第三，Rollback 或 Recovery。

业务里的回滚，不是 git revert。

很多业务动作一旦发生，没法真正回到过去。

所以更重要的不是事后假装能撤销。

而是前面先推演，没过就打回去，真出问题也要能恢复到稳定状态。

所以这些词不是 PPT 上的高级词。

它们应该从业务动作里长出来。

Agent 要进业务系统，不能只靠模型聪明。

还要靠这些让人能掌控它的产品结构。

## 2.7 第二章收束

【切页：第二章收束页】

第二章我想收束成一句话：

> 真正值得做的，不是把公司里的一切都规划成流程，也不是把一切都 Agent 化。

更现实的做法是：

先承认业务本来就是开放和混乱的。  
不要要求 Agent 一上来直接执行。  
先给关键业务动作一个沙盘。  
让方案先成形、被比较、被人判断。  
再决定哪些动作可以进入真实系统。

所以这里不是要把业务世界改造成代码世界。

很多业务之所以有生命力，正是因为它不是被完整规划出来的。

但 Agent 要进入这样的环境，至少需要一个缓冲层：

> 先把开放问题变成方案对象，  
> 再让人判断，  
> 再进入真实执行。

这就是 Amazon Canvas 给我的启发。

【切页：第三章占位】

第三章我会再把这个思路放回到我们自己的场景里：  
如果 Agent 进入协同办公笔记系统，会议、文档、任务和项目，应该怎么被组织成一个可以行动的业务空间。

# 第四章：留在台上

> 尽情使用它，探索它的边界

## 4.1 尽情使用它

【切页：尽情使用它】

最后一章，我想回到人自己。

面对 Agent，我越来越觉得，最危险的不是用错，而是一直站在岸边看。

很多事情你不真正把它放进自己的工作，就永远只能听别人讲它强或者不强。

但 Agent 的能力边界，很多时候不是看测评看出来的，是在真实任务里撞出来的。

今天我们投入的 token，不一定马上变成所谓的业务成果。

它可能先变成文章、代码、方案、原型。

甚至很多代码最后会被丢进垃圾桶，很多想法最后不会上线。

但这不完全是浪费。

因为它至少让想法更快变成了可以看、可以改、可以否定的东西。

原来一个想法可能停留在脑子里，或者停留在一句“有空再说”里。

现在它可以很快变成一版草稿、一段代码、一个页面、一个 demo。

它不一定对，但它出现得更早。

这件事本身就会改变我们的探索速度。

---

## 4.2 从投入到产物，再到成果

【切页：投入 / 产出 / 成果】

如果按投入、产出、成果来看，我觉得大部分企业或许还没有真正享受到 AI 红利。

投入已经发生了。

模型费用在花，token 在消耗，工具在接入，人也在学习。

产出也开始出现了。

文章、代码、方案、报告、原型、Review 结果，确实比以前出来得更快。

但这些产出还不等于成果。

成果要看什么？

要看它有没有进入真实流程。  
有没有被复用。  
有没有降低成本。  
有没有提升质量。  
有没有带来业务结果。  
有没有沉淀成组织能力。

所以短期里，我们会看到大量废稿、大量废代码、大量没有上线的探索。

这很正常。

AI 红利不是 token 自动兑换出来的。

它需要人把这些产物筛选出来，组织起来，沉淀成新的工作方法。

---

## 4.3 它不是拔插替换的零件

【切页：不是替换人，而是重组任务】

这里也要避免一个误解。

AI 并没有真正做到一对一地替换掉 30% 的员工。

这点大家应该能达成共识。

它确实在很多任务上已经比初级白领强。

比如写草稿、整理信息、生成代码、做初步分析。

但在另一些任务上，它又不如人类稳定。

尤其是涉及复杂语境、组织关系、风险判断、长期责任的时候。

所以它绝对不是一个可以直接拔插替换的零件。

更不可能简单地直接取代一家公司 10%、20% 甚至 30% 的人。

更真实的变化是：

任务会被重新拆开。  
人的工作重心会被重新分配。  
产物会暴增。  
流程会被重组。

AI 红利如果会发生，也不是简单从“减少多少人”开始。

而是从“同样的人，能不能组织出更多有效成果”开始。

---

## 4.4 探索它的边界

【切页：知道边界，内心才不会慌】

很多人面对 Agent 会慌，是因为不知道它的边界。

不知道它什么时候可靠。  
不知道它什么时候会突然跑偏。  
也不知道自己应该站在哪个位置。

但只要你真正用过足够多次，你会慢慢知道：

哪些任务可以放手。  
哪些任务必须拆小。  
哪些地方必须检查。  
哪些结果只能当草稿。  
哪些场景必须由人来判断。

所以边界不是恐惧的来源。

边界反而是安全感的来源。

你知道边界，内心才不会慌。

你知道它在哪些地方强，哪些地方弱，你才知道自己应该怎么站稳。

---

## 4.5 长期会变好，短期会阵痛

【切页：长期会变好，短期会阵痛】

我不想把这件事讲得太轻松。

长期看，Agent 会让很多工作变好。

但短期看，它一定会带来阵痛。

因为它不是多了一个工具，而是改变了人的工作重心。

有些经验会被重新定价。  
有些流程会被重新拆开。  
有些岗位边界会变得模糊。  
有些人的节奏会被迫加快。

企业也会经历一段很尴尬的时期：

投入很多，产物很多，但成果还不稳定。

管理者会发现，“用 AI”不等于“立刻减少人”。

个体也会发现，“会用 AI”不等于“可以不判断”。

这段时间不舒服，是正常的。

关键不是假装没有阵痛，而是在阵痛里建立新的判断力。

---

## 4.6 你准备好了吗

【切页：你准备好了吗】

所以最后我想问的不是：

> 你看好 Agent 吗？

我想问的是：

> 你准备好了吗？

不是准备好被它替代，而是准备好和它一起工作。

你有没有把 Agent 放进真实任务里？  
你知道它在你工作里的边界吗？  
你知道哪些事可以交给它，哪些事必须你来判断吗？  
你有没有开始重建自己的工作方法？

Agent 不会等我们完全理解它以后再改变世界。

它会一边变强，一边进入我们的工作。

所以最好的姿态不是观望，也不是恐慌。

而是上手、试错、校准、站稳。

尽情使用它，拥抱它，也探索它的边界。

长期会变好，短期会阵痛。

但真正重要的是：

> 留在台上。

留在台上，才有机会参与下一幕。
