# 第二章调研：从 Coding Agent 到业务 Agent

## 调研目标

第二章不再做术语解释，也不讲成一套完整的“业务工程化四层框架”。

这一章只讲清楚三件事：

1. 为什么 Coding Agent 发展最快。
2. 为什么业务 Agent 更难，因为业务常常在混乱中前行。
3. Amazon Seller Canvas 为什么是一个值得拆的业务 Agent 案例。

核心判断：

> Coding Agent 的成功，不是模型先征服了代码，而是代码世界先给 Agent 提供了一个稳定工作的工程空间。业务世界更开放、更混乱、更难回滚，所以业务 Agent 不能照搬 Coding Agent，而需要先把高风险动作放进可推演的沙盘。

## 一句话主线

代码世界天然具备四个条件：

- 可视化
- 相对封闭
- 可验证
- 可回滚

业务世界通常刚好相反：

- 信息分散
- 边界流动
- 判断依赖人、经验、时机和语境
- 后果外溢，很多动作不可轻易回滚

但这里不建议把业务讲成“必须被重新规划成四层架构”。更合适的表达是：

> 业务不是完美流程。很多业务就是在混乱中前行，甚至伟大不可被规划。Agent 进入业务系统时，问题不是消灭混乱，而是在关键动作前增加一个可推演、可比较、可确认的缓冲空间。

## 资料如何支撑这个判断

### 1. Agent 不是更会聊天，而是会围绕目标使用工具

主流资料对 Agent 的描述有一个共同点：Agent 会围绕目标使用工具、处理上下文、推进多步任务，并可能影响外部系统。

PPT 里可以这样讲：

> Chatbot 主要回答问题；Agent 会围绕目标使用工具，并推动状态变化。

参考来源：

- OpenAI agent guide：强调 Agent 通过工具获取上下文、执行动作、编排其他 Agent，并把工具分为 Data、Action、Orchestration。  
  <https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/>
- Anthropic：区分 workflow 和 agent，workflow 是预定义路径，agent 是模型动态决定流程和工具使用。  
  <https://www.anthropic.com/engineering/building-effective-agents>
- IBM：强调 AI agents 的 tool calling、工作流优化和任务拆解。  
  <https://www.ibm.com/think/topics/ai-agents>
- Google Cloud：强调 reasoning、planning、memory 和 autonomy。  
  <https://cloud.google.com/discover/what-are-ai-agents>

### 2. Tool calling 是从“内容错误”到“状态错误”的分界线

Function calling / tool calling 让模型连接外部系统：

- 模型接收用户请求、上下文和可用工具。
- 模型决定是否调用工具，以及用什么参数调用。
- 应用侧真正执行工具。
- 工具结果返回给模型。
- 模型继续生成回应或继续下一步工具调用。

PPT 里可以这样讲：

> Tool calling 让模型从“会说”变成“会做”。也正因为它会做，错误就可能从内容错误变成业务状态错误。

参考来源：

- OpenAI Function Calling  
  <https://platform.openai.com/docs/guides/function-calling>
- OpenAI Tools  
  <https://platform.openai.com/docs/guides/tools>

### 3. 代码世界不是简单，而是有工程空间

代码世界并不简单，但它已经被工程系统包裹：

| 条件 | 代码世界里的表现 |
|---|---|
| 可视化 | 代码、目录、依赖、接口、提交记录、review 历史 |
| 相对封闭 | 仓库、文档、规范作为输入；代码、测试、构建作为输出 |
| 可验证 | typecheck、test、build、CI、review |
| 可回滚 | branch、PR、revert、tag、回退发布 |

PPT 里可以这样讲：

> 代码世界并不简单，但它的复杂是一种被工程系统包裹过的复杂。

参考来源：

- GitHub Pull Request Reviews：reviewer 可以 comment、approve、request changes。  
  <https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/reviewing-changes-in-pull-requests/about-pull-request-reviews>
- GitHub Protected Branches：可要求 status checks、pull request reviews、code owner approval。  
  <https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/managing-protected-branches/about-protected-branches>
- Git revert：通过生成新的提交来撤销已有提交带来的改动。  
  <https://git-scm.com/docs/git-revert>

### 4. HITL / Checkpoint / Rollback 只作为支撑概念，不在 PPT 里重讲成术语课

这些概念仍然重要，但第二章不需要大费功夫展开。

- Human in the Loop：敏感动作前让人 approve / edit / reject。
- Checkpoint：执行过程中的状态锚点，支持暂停、恢复、审计。
- Rollback：恢复稳定状态，不一定是“撤销上一步”。

PPT 里只需要把它们自然放进 Amazon Canvas 的语境里：

> Amazon Canvas 的价值，是让高风险动作先在沙盘里成形、被比较、被人判断，再决定是否进入真实系统。

参考来源：

- LangChain Human-in-the-loop middleware  
  <https://docs.langchain.com/oss/python/langchain/human-in-the-loop>
- LangGraph Persistence  
  <https://docs.langchain.com/oss/python/langgraph/persistence>
- Microsoft Agent Framework Checkpoints

### 5. Amazon Seller Canvas 是第二章重点案例

Amazon 在 Seller Central 引入 Canvas，把聊天式 AI、业务数据、洞察、方案和行动建议组织在一个可交互工作区里。

这类 Canvas 的启发不在于“界面好看”，而在于：

- 把问题、方案和影响先对象化。
- 让用户在真实执行前探索、比较、审查。
- 让 Agent 先形成候选提案，而不是直接进入真实系统。
- 给高风险业务动作提供类似 branch 的缓冲层。

PPT 里可以这样讲：

> 对不可轻易回滚的业务场景，沙盘不是装饰，而是真实执行前的代理现实。

参考来源：

- Amazon Seller Canvas 介绍  
  <https://www.aboutamazon.com/news/innovation-at-amazon/amazon-sellers-canvas-artificial-intelligence>
- Amazon Seller Assistant agentic AI 介绍  
  <https://www.aboutamazon.com/news/innovation-at-amazon/seller-assistant-agentic-ai>

## 第二章推荐结构

### 2.1 为什么发展最快的是 Coding Agent

核心句：

> Coding Agent 的成功，不是模型先征服了代码，而是代码世界先给 Agent 提供了一个安全的工程空间。

### 2.2 业务世界为什么难很多

核心句：

> 业务世界不是完美流程，而是在混乱中前行。

### 2.3 没有控制空间，业务 Agent 会撞上现实

核心句：

> 业务 Agent 不能照搬 Coding Agent。它需要的不是一套完美规划，而是关键动作前的推演空间。

### 2.4 Amazon Seller Canvas

核心句：

> Amazon Canvas 的价值不是漂亮界面，而是把复杂业务先放进一个可推演、可比较、可审核的沙盘。

### 2.5 为什么“提升销量”会变成“加商品”

核心句：

> “提升销量”不是一个动作，而是一组增长杠杆。“加商品”不是跳题，而是选品扩张的一条候选路径。

### 2.6 Amazon 的四类业务场景

核心句：

> 它不是给一句建议，而是把业务问题拆成表现分析、营销优化、库存决策、新品发布这些可操作场景。

### 2.7 从 Amazon 得到的方法论

核心句：

> 开放问题进入沙盘，变成方案对象；方案对象经过推演和审核，再进入真实执行。

## 可直接放 PPT 的话

- 代码世界并不简单，但它的复杂是一种被工程系统包裹过的复杂。
- Coding Agent 的成功，不是模型先征服了代码，而是代码世界先给 Agent 提供了安全空间。
- 业务世界不是完美流程，而是在混乱中前行。
- 伟大不可被规划，所以业务 Agent 不能靠一套完美流程解决所有问题。
- Agent 进入业务系统，不是消灭混乱，而是在关键动作前增加一个可推演的缓冲空间。
- Amazon Canvas 的价值不是漂亮界面，而是业务沙盘。
- “提升销量”不是一个动作，而是一组增长杠杆。
- “加商品”不是跳题，而是选品扩张的一条候选路径。
- 真正的业务 Agent，不是给一句建议，而是组织数据、提出选项、展示影响、等待决策、再推进执行。
