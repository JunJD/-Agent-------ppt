# 《当 Agent 进入业务系统》

## 第一章节：我与 AI

这一章不建议写成“我做过哪些项目”的流水账。  
更好的主线是：

> 我与 AI 的关系，经历了从“模型入口”，到“模型拓展”，再到“Agent 出世”，最后到“掌控 Agent”的变化。

这样后面讲 Agent 进入业务系统时，听众会自然接受：你不是突然开始讲 Agent，而是一路看着 AI 从公共入口变成可接入能力，又从模型应用变成执行体。

## 推荐分法：4 个阶段

不是按年份硬切，而是按“AI 在协作关系中的职责变化”切。

如果参考 `Phase 1 / Phase 2 / Phase 3` 那种图，这一章最重要的不是项目细节，而是把经历抽象成几种**有代表性的协作范式**。

也就是说：

- 图片素材是证据
- 章节主线是抽象关系变化
- 每个阶段只保留一个最有代表性的变化

## Phase 1：模型入口

一句话：

> ChatGPT 让模型进入大众视野，API 让它成为我可以接进产品的能力。

时间锚点：

- ChatGPT 出世
- OpenAI API 开放
- 2023.05 自己接入 AI，做 Chat 网页

对应意义：

- 这个阶段 AI 还不是协作者
- 它先是一个公共入口，再变成开发者可调用的能力
- 对我来说，关键不是“体验了 ChatGPT”，而是“把模型接进自己的产品”

素材证据：

- `2023.5.11接入AI.PNG`

这一页可以讲的话：

> ChatGPT 出来之后，AI 不再只是论文和模型能力，而是第一次变成所有人都能理解的入口。API 开放之后，它又从一个聊天产品变成开发者可以接入的能力。2023 年 5 月，我把它接进了自己的 Chat 网页，这才是我和 AI 关系真正开始变化的地方。

## Phase 2：模型拓展

一句话：

> AI 模型在变成 Agent 之前，先通过外部知识、行为约束、工具能力和业务场景变得更可用。

时间锚点：

- 2023.07 前后

总图：模型拓展的四类方式

- 知识拓展：RAG、embedding、向量检索，把外部知识接到模型上下文。
- 行为拓展：prompt、结构化输出、fine-tuning，让模型输出更稳定、更贴合任务。
- 能力拓展：function calling / tool calling，让模型连接外部系统、数据和动作。
- 场景拓展：把模型嵌入搜索、客服、代码审核、办公流等真实业务流程。

分图一：拓展 AI 的知识面

- embedding
- 向量化检索
- 拓展模型原本不知道的外部知识

分图二：拓展 AI 的应用场景

- 接入 GitHub AI 审核
- 让 AI 进入代码审核这样的真实流程
- 从生成内容，开始参与判断

对应意义：

- 向量化检索和 GitHub AI 审核都属于大模型应用，还不是 Agent
- 总图讲“模型怎么被拓展”
- 知识面拓展解决“AI 知道什么”
- 应用场景拓展解决“AI 能被放到什么场景里”
- 人仍然是主流程，AI 是被嵌入的增强能力

素材证据：

- `解除向量化检索拓展AI知识面.jpg`
- `2023.7.15github接入AI审核.JPG`

页面建议：

- `2-0 模型拓展总图`：不放个人截图，讲知识、行为、能力、场景四类拓展。
- `2-1 知识面拓展`：放向量化检索素材。
- `2-2 应用场景拓展`：放 GitHub AI 审核素材。

这一页可以讲的话：

> 很快我就发现，单纯聊天没有那么大意思。模型要进入真实问题，先要被拓展：有些拓展发生在知识层，比如 embedding 和向量检索；有些拓展发生在场景层，比如把 AI 放进 GitHub 审核。这里其实是两个不同故事：一个是拓展 AI 知道什么，一个是拓展 AI 能被用在哪里。

## Phase 3：Agent 出世

一句话：

> Agent 不是 Manus 之后才出现；更早的 AutoGPT 式研究已经在探索让 AI 长时间工作。

时间锚点：

- AutoGPT 式长任务研究
- 2024.03 起广泛使用 Cursor / coding agent / vibe coding
- 2025 Manus 把 Agent 概念带火

代表动作：

- 早期 Agent 的核心是：分任务、执行、验收、再执行
- 当时还没有今天成熟的 tool / tool calling 产品概念
- 我在 2024 已经用 Cursor / coding agent 做 vibe coding
- 通过页面、代码仓库和体验来验证 Agent 的结果

对应意义：

- Agent 的分水岭不是“会不会聊天”
- 而是“能不能接收任务并推进到可验证结果”
- Manus 是概念扩散点，不是实践起点
- 这个阶段更像“让 AI 长时间工作”的范式出世

素材证据：

- `IMG_8023.PNG`
- 2024 vibe 出来的 `alading` GitHub repo 截图
- 早期开源 / 公司内部 coding agent 使用截图

这一页可以讲的话：

> Agent 不是 Manus 火了以后才出现的。更早的 AutoGPT 研究就已经在尝试让 AI 长时间工作：先分任务，再执行，再验收，再继续执行。那时还没有今天这么明确的 tool 概念，但 Agent 的形态已经出来了。到 2024 年，我已经在用 Cursor 和 coding agent 做 vibe coding，很多项目不是看它怎么回答，而是看它最后做出来的页面和代码能不能成立。

## Phase 4：掌控 Agent

一句话：

> 到 Codex 阶段，重点不是让 Agent 一次改一个巨大任务，而是按重要程度和执行复杂度选择掌控方式。

时间锚点：

- Codex 阶段 / 当前阶段

代表动作：

- 公司级重点项目：重要程度高、执行复杂度高，四个项目 / 多 repo / 多 Agent 并行，但每个任务颗粒度都很小，小到能验收代码和页面
- 开源维护项目：作为一个案例点，用主动触发架构 review、code review、BDD 行为测试，让 review 和回归机制参与掌控质量
- 简单交付项目：执行复杂度低，给别人做项目时采用 spec-kit 式开发，不看一行代码，只管最终结果

对应意义：

- 人开始从“实现者”转成“任务切分者 + 验收者 + 风险控制者”
- 掌控感不是来自盯住每一行代码，而是来自选择了正确的控制策略
- 高重要、高复杂项目靠拆小任务和持续验收
- 需要持续维护的项目靠 review / 架构 / BDD 回归机制
- 低复杂交付项目可以只看最终结果

总图：掌控 Agent 的 2x2 Prioritization Matrix

- 横轴：执行复杂度，从低到高
- 纵轴：重要程度，从低到高
- 左上：Quick Win，高重要、低复杂，轻量执行，结果验收
- 右上：重点项目，高重要、高复杂，拆小任务，多 Agent 并行，持续验收
- 左下：低优先填充，低重要、低复杂，自动化或顺手处理
- 右下：先拆解，低重要、高复杂，不直接让 Agent 狂奔
- 公司级项目、开源项目、简单交付只是案例点，不直接定义象限

三个展开场景：

### 场景一：公司级重点项目

核心行为：

> 它落在高重要、高执行复杂度区域：一个项目可能有四个 repo，可以开四个 Agent 快速并行，但每个 Agent 的任务都要足够小，小到能被我验收。

这一页要表达：

- 公司级项目重要程度高，执行复杂度也高
- 多 repo / 多 Agent 并行是效率来源
- 小任务颗粒度是掌控来源
- 每个任务都要能看代码，也要能看页面

### 场景二：开源维护项目

核心行为：

> 开源项目不先固定到某个象限，但只要它需要持续维护，就要用主动触发的 skill 和回归机制来掌控质量。

这一页要表达：

- 这是另一种范式：不是每次都亲自从头审完
- 开发完成后触发 code review 找问题
- 触发架构 review 看结构是否需要重构
- 触发 BDD 行为测试 / dev3000 回归，验证行为没有坏
- 掌控来自机制，而不是只靠人工盯

### 场景三：简单交付项目

核心行为：

> 给别人做项目时，我可以采用 spec-kit 式开发，不看一行代码，只管最终结果。

这一页要表达：

- 先把需求规格和验收标准写清楚
- 中间过程可以放手
- 不追求逐行代码质量
- 最终只看页面、体验、交付结果是否成立

素材证据：

- 公司级多 repo / 多 Agent / 小任务验收截图
- 开源项目架构 review / code review / BDD 行为测试 / dev3000 回归截图
- 给别人做项目的 spec-kit / 最终结果截图

这一页可以讲的话：

> 到 Codex 这一阶段，我更关心的不是“Agent 能不能一次做一个巨大任务”，而是“我能不能根据重要程度和执行复杂度掌控它”。公司级重点项目，我会多 repo、多 Agent 并行，但任务必须足够小；开源维护项目，我会让 code review、架构 review、BDD 回归参与质量控制；简单交付项目，我可以用 spec-kit 式开发，不看一行代码，只看最终结果是否成立。

## 这一章的收束

推荐转场：

> 很多程序员其实已经和我一样，全面拥抱 Agent 了。  
> 这不是因为程序员更愿意冒险，而是因为代码世界天然有一套控制系统：分支、diff、review、测试、版本和回滚。  
> 所以下面我们先认识一下，代码世界是怎么接住 Agent 的：它依靠哪些控制系统，又形成了哪些体系语言。

继续往下讲，就可以自然进入第二章：

- 什么是可控的 Agent
- 什么是 human in the loop
- 什么是 checkpoint
- 什么是 rollback
- 为什么 Agent 进入业务系统后，不能只谈“更智能”，还必须谈“可控”
