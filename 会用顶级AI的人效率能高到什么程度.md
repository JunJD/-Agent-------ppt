# 会用顶级 AI 的人，效率能高到什么程度？

> 一个 Coding Agent 实践者的四象限框架

![会用顶级 AI 的人，效率能高到什么程度？](./article-images/01-cover-efficiency.png)

过去一年，社区里关于 Coding Agent 的文章越来越多。

有人讲 Prompt 怎么写，有人讲怎么让 AI 改一个页面，有人讲怎么生成测试、补文档、修 bug。这些当然有价值，但在我看来，大多数实践还停留在一个阶段：

> 把 AI 当成一个更快的程序员。

而我现在更关心的问题是：

> 如果一个人真的会用顶级 AI，他的效率边界会被推到什么程度？

我的答案不是某个神奇工具，也不是某条万能 Prompt，而是一套任务分型方法。

我会先把 Coding 任务按两个维度拆开：

- 重要程度
- 执行复杂度

这两个维度组合起来，就得到一个 Coding Agent 的掌控四象限。

## 先看总图：不是所有任务都该用同一种 AI 协作方式

![Agent 掌控策略 2x2](./article-images/02-agent-control-quadrant.png)

这套分类不是为了判断“哪个任务更高级”，而是为了回答一个更关键的问题：

> 这个任务应该用什么方式掌控 Agent？

同样是让 AI 写代码，不同象限里的协作方式完全不同。

![任务路由](./article-images/03-task-routing.png)

真正的效率提升，不是“让 AI 多写代码”，而是：

> 先判断任务类型，再选择对应的协作方式。

## 为什么普通的 AI 编程实践不够

很多人用 Coding Agent 的方式是这样的：

> 我有一个需求，丢给 AI；AI 写完代码，人看结果；能用就提交，不能用就继续让它改。

这种方式在小任务上没有问题。

但一旦任务进入公司项目、跨仓库项目、长期维护项目，它就会开始失控。因为 Agent 不天然知道什么重要，不天然理解业务边界，也不天然知道某个改动会影响哪些长期演进。

所以我的实践方式变成了另一种：

![普通用法 vs 掌控式用法](./article-images/04-naive-vs-controlled.png)

人不是消失了，而是位置变了。

过去人是实现者。现在更像是：

- 任务切分者
- 边界定义者
- 风险判断者
- 结果验收者

## 第一类：公司级重点项目

公司级项目通常落在右上角：

> 高重要程度，高执行复杂度。

这类项目不能靠一句“帮我实现一下”解决。它需要人先把边界拆清楚，再让 Agent 在清晰边界里高速执行。

我负责过一个知识库 Agent 相关项目，涉及多个仓库：

- 知识库本身
- AgentKit
- 富文本输入框
- 笔记编辑器
- 笔记相关模块

过去这类需求推进起来很重，因为一个功能可能穿透多个仓库。现在我的做法是，把相关仓库同时打开，并行使用 Coding Agent。

![公司级项目：多 repo 并行](./article-images/05-company-multirepo.png)

公司级项目的关键，不是“开了几个 Agent”，而是任务必须拆到足够小。

小到什么程度？

![任务颗粒度](./article-images/06-task-granularity.png)

我的标准很直接：

> 如果我自己都不知道边界在哪里，就不要假装 Agent 会知道。

在公司级项目里，效率来自并行，掌控来自拆分。

如果项目没有多个仓库，也可以用多个 worktree 做类似的并行开发。重点不是形式，而是把复杂任务拆成多个可验证的局部。

## 第二类：开源兴趣项目

开源兴趣项目通常没有公司项目那么重要，但复杂度并不低。

它可能有产品形态，有竞品，有架构，有前后端，有体验细节。但我不可能像公司项目一样投入大量时间。

所以这类项目的目标不是“无限打磨”，而是：

> 用机制压缩探索成本和验证成本。

我的流程大概是这样：

1. 让 AI 调研竞品和社区方案。
2. 人确定产品框架和架构方向。
3. Agent 实现功能。
4. 用 Code Review、架构 Review、BDD 行为测试回收问题。
5. 人只重点判断风险点和不确定点。

![开源兴趣项目：用机制回收质量](./article-images/07-open-source-quality-loop.png)

以 Openwork 这类兴趣项目为例，前期更适合先把产品形态、界面层级和核心交互拆出来。不是一上来就让 Agent 狂写代码，而是先把“这个产品到底长什么样、哪些区域承担什么职责”讲清楚。

![Openwork 架构拆解](./会用AI的人/架构拆解.png)

这类项目的重点，不是逐行检查 AI 写出的所有代码。

Review 不是为了让 AI 审完就直接提交。真正有价值的是它指出的：

- 不确定点
- 风险点
- 设计模糊点
- 行为不一致点

人再去判断这些点是否真实存在，是否值得修，是否影响下一步。

质量回收里，BDD 行为测试尤其关键。

以 Openwork 这类桌面 Agent 项目为例，BDD 可以沉淀成一组稳定的 `.feature` 行为契约：

- `app-launch.feature`：验证桌面应用启动、Launcher 路由、窗口切换、搜索结果和历史线程打开。
- `agent.feature`：验证 Agent 长任务的流式返回、连续多轮、取消、暂停、审批恢复。
- `workspace.feature`：验证全局 workspace、线程 workspace、重启后的持久化、越界读取拒绝。
- `tool-approval.feature`：验证写文件审批里“新增/修改”的判断，以及审批展示信息和真实工具参数分开保存。
- `external-links.feature`：验证公共 https 可以打开，但 localhost/private network 会被拒绝。
- `model-provider.feature`：验证模型供应商远程列表失败后的错误状态不会被刷新冲掉。

这些 `.feature` 文件给我的启发是：BDD 不是把代码逻辑翻译成自然语言，也不是给所有函数都写一层“故事”。它真正适合锁住的是：

- 用户可见工作流
- 跨进程协作
- 窗口生命周期
- IPC 契约
- 持久化状态
- 安全边界
- Agent 暂停、恢复、审批这类长流程

好的 `.feature` 场景通常很小，但边界很清楚：

![BDD .feature：写行为契约](./article-images/08b-bdd-feature-files.png)

它的结构基本就是：

```gherkin
功能: 某个产品能力的行为契约
  为了避免某类用户路径或系统边界被改坏
  作为这个模块的维护者
  我需要关键行为在重构后仍然稳定

  场景: 用户完成关键动作后状态正确
    假如 初始状态存在
    当 用户执行一个动作
    那么 可观察结果成立
    而且 不破坏关键边界
```

这个模板看起来简单，但它背后有一个很重要的判断：**BDD 测的是行为，不是实现。**

Openwork 里的 `workspace.feature` 不关心底层怎么存路径，而是关心“设置全局 workspace 后能不能再次读到”“线程 workspace 能不能覆盖全局 workspace”“读取 workspace 外的路径会不会被拒绝”。这就是行为契约。

`agent.feature` 也不是去测某个 runtime 函数，而是测 Agent 长任务的关键生命周期：能不能流式完成、能不能取消、HITL 暂停后能不能批准或拒绝并恢复。

这种测试很适合开源兴趣项目，因为它可以把质量回收从“我每次人工全量体验一遍”变成“我把关键行为写成可重复触发的门禁”。

对于后端，BDD 行为测试通常比较直接：围绕 feature 写场景，生成测试脚本，再执行验证。

对于前端，我更看重真实界面和真实交互：启动页面、自动操作关键路径、截图或录屏，再检查布局和交互是否符合预期。

![BDD：后端看行为，前端看路径](./article-images/08-bdd-front-back.png)

前端行为验收不能只停留在“代码已经实现”。更可靠的方式，是把关键用户路径跑通，并保留截图或录屏作为结果证据。

![Openwork Launcher 搜索结果](./会用AI的人/实现截图1.png)

![Openwork Agent 对话界面](./会用AI的人/实现截图2.png)

这类项目的关键不是把 AI 当外包，而是建立一套质量回收机制。

## 第三类：小产品、小 Demo、小玩具

还有一类任务，不重要，也不复杂。

这类任务常见于和朋友一起做小产品、小工具、小 Demo。它的目标不是长期维护，而是快速把想法变成可体验的东西。

这类任务最适合两种方式：

- SpaceKit / SpecKit 式开发
- vibe coding 式开发

SpaceKit / SpecKit 式开发更像是：先和 AI 把想法聊清楚，沉淀需求、规范和任务，再让 Agent 按规格实现。人不盯过程，只验收最终体验。

vibe coding 更随意：想到一个点子，直接说给 AI；AI 改页面、改交互；人体验一下，再决定要不要继续改。

![小产品 / 小 Demo：快速循环](./article-images/09-demo-fast-loop.png)

这类任务不需要太重的工程化动作。它的价值在于极低成本地把想法变成可体验的东西。

很多过去只会停留在脑子里的念头，现在可以在半小时、一小时内变成一个真实页面。

它未必完美，但它已经足够支持一个判断：

> 这个想法有没有继续做的价值？

## 三套实践方式，对应三种掌控模型

三类实践放在一起，会得到这样一张图：

![三套实践，对应三种掌控模型](./article-images/10-three-practice-models.png)

也可以换一个角度看：

- 重要且复杂：人负责拆分和验收，AI 负责并行执行。
- 不重要但复杂：人负责方向，AI 负责调研、实现、Review 和验证。
- 不重要且不复杂：人负责想法，AI 负责快速变成可体验结果。
- 重要但不复杂：人负责规格，AI 负责快速交付。

## 会用 AI 以后，人到底变强在哪里？

表面上看，AI 提升的是写代码速度。

但我真正感受到的变化不是写代码更快，而是自己的工作方式被重构了。

过去一个人只能顺序推进：

> 理解需求 → 写代码 → 自测 → 修问题 → 提交。

现在可以变成：

![从线性工作，到并行组织](./article-images/11-linear-to-parallel.png)

这不是简单快 30%、50%，甚至不是快一倍。

真正的变化是：

> 人可以同时推进更多层次的事情。

公司复杂项目里，AI 扩大执行半径。

个人开源项目里，AI 压缩探索成本。

小产品和 Demo 里，AI 释放想法密度。

## 最后：AI 的边界，决定人的新能力

AI 能写代码，但它不天然知道什么重要。

AI 能拆任务，但它不天然理解业务风险。

AI 能做 Review，但它不天然拥有最终判断权。

所以会用顶级 AI 的人，真正强的不是会写某条 Prompt，而是具备一组新的工作能力：

![AI 的边界，决定人的新能力](./article-images/12-human-capability-boundary.png)

任意一种和 AI 协作的方式，都会提高效率。

但真正拉开差距的，是人能不能持续和它融合，持续找到它的边界，然后把自己训练成那个能驾驭边界的人。

这才是我理解的答案：

> 会用顶级 AI 的人，效率高，不是因为他把自己变成了十个程序员。
>
> 而是因为他开始像一个小型组织一样工作。
