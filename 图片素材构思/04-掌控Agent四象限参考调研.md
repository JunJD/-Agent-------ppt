# 素材 04：掌控 Agent 2x2 参考调研

## 结论

第四阶段的总图建议直接借用 `2x2 Prioritization Matrix` 的经典视觉语言。

但不要把它说成“优先级矩阵”本身，而是改造成：

> Agent 掌控策略 2x2

它不是在回答“先做哪个项目”，而是在回答：

> 这个项目应该用什么方式掌控 Agent？

## 为什么不用“复杂度 × 长期维护”

`复杂度 × 长期维护` 可以解释一部分场景，但它会带来一个问题：

- 听众会以为四个象限就是四种项目类型。
- 你的开源项目会被硬放进左上角。
- 但真实情况是：开源项目只是一个案例点，它的位置可以随项目重要性和复杂度变化。

所以更稳妥的方式是：

> 先用通用的 2x2 Prioritization Matrix 表达判断框架，再把你的三个场景作为案例点贴进去。

## 推荐坐标

横轴：

> 执行复杂度

纵轴：

> 重要程度

这和常见的 `Impact × Effort` / `Value × Complexity` 矩阵接近。

## 四个通用象限

### 左上：Quick Win

含义：

- 高重要
- 低执行复杂度

Agent 掌控方式：

- 轻量执行
- 快速验收
- 结果导向

可放案例：

- 简单但重要的交付项目
- spec-kit 式开发也可能靠近这里

### 右上：重点项目

含义：

- 高重要
- 高执行复杂度

Agent 掌控方式：

- 拆小任务
- 多 Agent / 多 repo 并行
- 代码验收 + 页面验收
- 持续 review

可放案例：

- 公司级多 repo 项目

### 左下：低优先填充

含义：

- 低重要
- 低执行复杂度

Agent 掌控方式：

- 自动化
- 顺手处理
- 批量处理

可放案例：

- 低风险小优化
- 文档整理
- 非关键辅助任务

### 右下：先拆解

含义：

- 低重要
- 高执行复杂度

Agent 掌控方式：

- 不直接交给 Agent 狂奔
- 先拆解
- 降低复杂度
- 或暂时不做

这是这张图里最有判断力的象限：

> 不是所有事情都应该交给 Agent 做。复杂但不重要的任务，先拆、后做，甚至不做。

## 三个个人场景如何放

### 公司级多 repo 项目

建议位置：

> 右上：重点项目

原因：

- 重要程度高
- 执行复杂度高
- 需要多 repo / 多 Agent 并行
- 但每个任务颗粒度必须小

表达方式：

> 高重要、高复杂时，掌控来自拆小任务和持续验收。

### 开源维护项目

建议位置：

> 不强行固定一个象限，作为案例点贴在中上或右上附近。

原因：

- 它是否高复杂，取决于具体改动。
- 它是否高重要，取决于这次 release / 用户影响。
- 它的核心不是象限位置，而是掌控机制：架构 review、code review、BDD 行为测试、dev3000 回归。

表达方式：

> 开源维护不靠一次性生成结果，而靠可反复触发的质量机制。

### 简单交付项目

建议位置：

> 左上或左下边界，取决于项目重要性。

原因：

- 执行复杂度低
- 是否重要看交付对象和场景
- 掌控方式可以更结果导向

表达方式：

> 低复杂交付可以 spec-kit 式开发：规格清楚、过程放手、结果验收。

## 推荐一句话

> 掌控 Agent 的关键，不是让它做多大的任务，而是先判断任务的重要程度和执行复杂度，再选择验收方式。

## 推荐图名

- Agent 掌控策略 2x2
- 掌控 Agent：重要程度 × 执行复杂度
- 不同项目，不同 Agent 掌控方式
