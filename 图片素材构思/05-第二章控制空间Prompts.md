# 素材 05：第二章控制空间 Prompts

## 使用约定

- 用于第二章《控制空间：为什么 Coding Agent 先成立》。
- 这一章现在按更口语的顺序讲：
  - Coding Agent 为什么先跑出来。
  - 业务不是问一句，就能办。
  - 协同文档前时代，最痛苦的是合并。
  - 协同文档怎么把“人想怎么改”变成有限 action。
  - Amazon Canvas 怎么把“销量怎么上去”先放到画布里看。
  - Human in the Loop 不是每步点确认，而是最后真动系统前，人能看懂、能改、能说不。
- 统一使用 `guizang-ppt-skill` 的“电子杂志 × 电子墨水”方向。
- 统一强调色：vivo 蓝 `#415fff`。
- 所有图片默认 16:9 横向构图。
- 图中文字使用中文，标签短一点，像人说话。
- 不生成假产品 UI、不生成假 logo、不画机器人、不画巨大 AI 大脑、不画握手隐喻。

## 章节视觉主线

第二章要讲得土一点：

> 代码世界能让 Agent 先跑出来，是因为它有仓库、diff、测试、回滚。业务世界麻烦在：用户说的是一句人话，但系统真要做事，必须落到具体动作。协同文档给了一个启发：别穷举人怎么改，先把操作变成有限 action。Amazon Canvas 也是类似：先别真动业务，先放画布里摊开，人看完再执行。

建议视觉顺序：

- `01 Coding Agent 工作空间`：代码世界有仓库、diff、测试、回滚。
- `02 业务不是问一句就能办`：用“销量上不去”拆“要啥 / 给啥 / 怎么算”。
- `03 协同文档前时代`：附件乱飞、版本打架、人工合并。
- `04 协同文档怎么解这个事`：人想怎么改都行，系统先变成几种 action。
- `05 Amazon Canvas 先别真动业务`：先摊开、先比较、人点头、再执行。
- `06 人不是每步点确认`：不是确认按钮，而是守住最后那一步。

- 以前一起改文档，最烦的是合并。

- 人不是每步点确认，而是真执行前能看懂、能改、能说不。
