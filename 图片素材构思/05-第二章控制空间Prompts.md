# 素材 05：第二章控制空间 Prompts

## 使用约定

- 用于第二章《控制空间：从 Coding Agent 到业务 Agent》。
- 这一章不大费功夫讲完整框架，只讲清楚三件事：
  - Coding Agent 为什么先成立。
  - 业务世界为什么复杂、混乱、难回滚。
  - Amazon Seller Canvas 如何把开放问题放进沙盘。
- 统一使用 `guizang-ppt-skill` 的“电子杂志 × 电子墨水”方向。
- 统一强调色：vivo 蓝 `#415fff`。
- 所有图片默认 16:9 横向构图。
- 图中文字使用中文，标签短而清晰。
- 不生成假产品 UI、不生成假 logo、不画机器人、不画巨大 AI 大脑、不画握手隐喻。

## 章节视觉主线

第二章不是术语解释，也不是完整的业务工程框架，而是一个判断：

> Coding Agent 的成功，不是模型先征服了代码，而是代码世界先给 Agent 提供了一个稳定工作的工程空间。业务世界更混乱，不能被强行规划成完美流程，所以高风险动作需要先进入沙盘。

建议视觉顺序：

- `01 Coding Agent 工作空间`：代码世界天然具备可视化、相对封闭、可验证、可回滚。
- `02 业务世界为什么难`：信息分散、边界流动、判断依赖人、后果外溢。
- `03 Amazon 沙盘 / Canvas`：高风险动作先推演、再审核、再执行。
- `04 Amazon 增长杠杆拆解`：解释“提升销量”为什么会走到补货、营销、新品。
- `05 Amazon Canvas 方案对象`：解释 Canvas 如何把开放问题变成可比较方案。

## 放进 PPT 时的页面文案建议

- Coding Agent 的成功，不是模型先征服代码
- 代码世界先提供了安全工程空间
- 业务世界不是完美流程，而是在混乱中前行
- 业务 Agent 不能照搬 Coding Agent
- Amazon Canvas 的价值不是漂亮界面，而是业务沙盘
- “提升销量”不是一个动作，而是一组增长杠杆
- “加商品”不是跳题，而是选品扩张的一条候选路径
- Amazon Canvas 的价值是把开放问题变成方案对象
