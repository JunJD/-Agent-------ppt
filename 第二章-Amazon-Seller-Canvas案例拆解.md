# 第二章调研：Amazon Seller Canvas 案例拆解

## 核心判断

Amazon Seller Canvas 值得讲的点，不是“它做了一个漂亮的 AI 页面”，而是它把 Seller Assistant 从聊天入口推进到了业务控制空间：

> 卖家提出开放问题，系统把它转成可视化、可比较、可推演、可审批、可执行的业务方案对象。

这正好承接第二章：

- 业务环境原本开放、分散、不可轻易回滚。
- Canvas 把问题先放进一个代理现实里。
- Seller Assistant 不急着直接改真实系统，而是先组织数据、洞察、选项、预测、风险和下一步动作。

## 先回答关键疑问：为什么“提升销量”会变成“加商品”

这不是跑偏，而是业务目标被拆成增长杠杆。

“提升销量”不是一个可直接执行的动作，它至少可能落到几条路径：

- 库存：热销商品是否缺货，是否需要补货或调拨？
- 营销：广告预算、曝光、转化、促销是否需要调整？
- 商品页：listing、A+ 内容、图片、标题是否需要优化？
- 价格：是否需要折扣、调价、清库存？
- 选品：是否有相邻品类、变体或新品线可以扩张？
- 履约：是否需要优化 FBA、仓储、配送速度和库存成本？

所以当问题是“怎么提升销量”时，Canvas 的价值不是给一个口号式建议，而是先诊断：

1. 当前销量问题来自哪里？
2. 哪些增长杠杆可用？
3. 每个杠杆的收益、风险、成本、时间是什么？
4. 哪些动作可以现在做，哪些动作要先做沙盘？

“加商品 / 新品发布”只是增长杠杆之一。它通常不是第一步直接执行，而是当系统看到历史销售趋势、客户洞察、类目需求信号、竞争强度和当前目录相邻机会后，把它作为一个候选策略放进 Canvas。

## Amazon 文章里实际给了四类场景

### 1. 业务表现分析

卖家可以问：

- Can you analyze my sales performance?
- How are my products performing?

Canvas 做的不是简单回答，而是把销售数据、客户流量、趋势和行动建议放到同一个视图。

这里的关键是：

- 把散落数据组织成一个业务视图。
- 主动发现异常或机会。
- 例如某个商品销量突然上升，就给出上下文和补库存等建议。

这对应第二章里的“可视化层”。

### 2. 营销表现优化

卖家可以问：

- How can my promotional campaigns perform better?

Canvas 会分析：

- campaign spend
- impressions
- conversions
- product-level sales lift

然后不是给泛泛建议，而是提出多种前瞻策略，说明每个策略的理由和预期结果。卖家还可以加约束，比如：

- 这个月要降低花费
- 只关注库存过剩商品

Canvas 会根据约束实时调整计划。

这里的关键是：

- 不是一次性答案，而是交互式规划。
- 约束进入系统，方案随之重算。
- 策略有理由、有预测、有可比较性。

这对应第二章里的“验证层”和“沙盘层”。

### 3. 库存决策

卖家可以问：

- What products should I restock?

Canvas 不只是列一个补货清单，而是提供多条路径：

- 现在补货
- 延迟补货，继续观察需求
- 给库存过剩商品打折

同时展示每个选项对这些指标的影响：

- revenue
- cash flow
- stockout risk
- storage fees
- competitive positioning

卖家还可以做 what-if：

- 如果需求下降 10% 呢？
- 如果我打折而不是补货呢？

这里的关键是：

- 不直接替卖家下单。
- 先做情景推演。
- 把不可逆或高成本决策变成可比较方案。

这对应第二章里的“回滚前置”：事后回滚难，所以事前要推演。

### 4. 新品发布 / 商品扩张

卖家可以问：

- What product should I launch next?

Canvas 会结合：

- historical sales trends
- customer insights
- category demand signals
- competitive intensity

然后形成优先级策略，而不是单一推荐：

- 当前目录的相邻变体
- 进入新类目
- 不同扩张路径的投资、风险和盈利时间

当卖家准备推进时，Seller Assistant 再把规划转成执行准备：

- launch checklist
- initial inventory forecast
- staged launch actions

这里就是用户问的“怎么从提升销量变成加商品”的关键：

> 新品发布不是聊天里的跳跃答案，而是 Canvas 在增长诊断后生成的一类扩张策略。它先以方案对象存在，经过比较、推演和确认，再进入执行准备。

## 这套产品的真正结构

可以拆成六步：

1. 开放问题  
   卖家问“怎么提升销量”“广告怎么优化”“该补哪些货”“下一步 launch 什么产品”。

2. 数据汇聚  
   系统拉取销售、流量、广告、库存、类目、竞争、客户洞察等数据。

3. 机会识别  
   系统识别增长机会、风险点、异常波动和约束条件。

4. 方案生成  
   不是单一答案，而是多个可比较选项。

5. 沙盘推演  
   展示收益、成本、风险、现金流、库存、时间线等影响。

6. 分阶段执行  
   用户确认后，Seller Assistant 才进入 checklist、forecast、staging、更新价格、创建补货订单等动作。

## 为什么这是业务 Agent 的好案例

它完整体现了第二章的控制空间：

| 第二章概念 | Seller Canvas 对应做法 |
|---|---|
| 可视化 | 个性化画布聚合数据、洞察、推荐动作 |
| 封闭层 | 把开放问题收敛到营销、库存、新品等场景 |
| 验证层 | 用预测、约束、指标影响和 what-if 场景辅助判断 |
| 回滚层 | 高风险动作先在 Canvas 里推演，减少直接打到真实系统 |
| 产物化 | 形成策略、方案、清单、预测、待执行动作 |
| Human in the Loop | 卖家在关键路径上选择、约束、确认 |
| Agent 执行 | Seller Assistant 在授权后推进价格更新、补货订单、新品发布准备等动作 |

## 对第二章的建议主线

第二章后半段可以叫：

> 业务 Agent 案例：Amazon Seller Canvas 如何把增长问题变成可执行方案

建议讲法：

1. 先抛出疑问：为什么问“提升销量”，系统会讲到补货、营销、甚至新品发布？
2. 解释：因为销量增长不是一个动作，而是一组业务杠杆。
3. 拆 Amazon 的四类场景：绩效、营销、库存、新品。
4. 强调 Canvas 的价值：不是聊天回答，而是把问题转成方案对象。
5. 得出方法论：业务 Agent 的核心不是多聪明，而是能把开放问题收敛成可验证、可比较、可执行的控制空间。
6. 再迁移到我们的协同办公笔记系统：会议和文档里的开放问题，也要先变成任务、方案、审批、执行记录。

## 可直接放 PPT 的话

- “提升销量”不是一个动作，而是一组增长杠杆。
- Canvas 的价值不是回答问题，而是把开放问题变成可比较的方案对象。
- 它不是先执行再观察，而是先推演、再审核、再执行。
- “加商品”不是跳题，而是销量增长在选品扩张上的一个候选路径。
- 真正的业务 Agent，不是给一句建议，而是组织数据、提出选项、展示影响、等待决策、再推进执行。
- 业务系统要学 Amazon 的不是界面，而是把高风险决策放进沙盘。
